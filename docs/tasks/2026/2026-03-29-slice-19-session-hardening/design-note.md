## Design Note

- Problem:
  - Slice 18 already proved remote desktop login and run control, but the remote access token still lived only in memory. Desktop restart always discarded the session, and the connection surface could not distinguish auth-invalid restores from transport-only outages.
- Goal:
  - Add a desktop-only secure session cache and restore path that keeps remote sign-in durable across restarts while preserving current shared contracts and remote-hub auth semantics.
- Acceptance Criteria:
  - Cached remote session restore runs before desktop default-route resolution.
  - Restore refuses expired or profile-mismatched entries locally before any network call.
  - Auth-invalid revalidation clears the cache and returns the user to `/connections`.
  - Transport-only revalidation preserves the cached session summary and remembered project route, but the hub surface becomes explicitly read-only / degraded.
  - Secure-store unavailability degrades to memory-only behavior with a user-visible warning.
- Non-functional Constraints:
  - No `schemas/` changes.
  - No remote-hub route or auth DTO changes.
  - `projectId` stays in the existing local profile store and is never moved into the secure vault.
- MVP Boundary:
  - One persisted remote-session record keyed to the active desktop profile binding.
  - One Tauri wrapper module that exposes app-local load/save/clear hooks to the desktop store.
  - One startup initializer path in desktop bootstrap.
- Layer Placement:
  - `apps/desktop/src-tauri/src/session_vault.rs` owns serialization, secure-store fallback semantics, local expiry checks, and profile-binding validation.
  - `apps/desktop/src-tauri/src/lib.rs` owns Tauri command exposure and state wiring for the secure cache service.
  - `apps/desktop/src/stores/connection.ts` owns session restore orchestration, cache synchronization during login/logout/profile changes, and warning propagation.
  - `apps/desktop/src/main.ts` owns boot ordering so restore happens before route creation.
  - `apps/desktop/src/views/ConnectionsView.vue` owns the degraded/read-only and memory-only warning copy.
- Module Boundaries:
  - The secure session cache is app-local to desktop/Tauri and must not leak into shared client or schema modules.
  - Existing `HubClient` and remote auth clients remain authoritative for network validation; the secure cache only seeds restore.
  - Local profile storage still owns `mode`, `baseUrl`, `workspaceId`, `email`, and `projectId`.
- Inputs:
  - Existing desktop connection profile persistence, remote auth client `login / logout / getCurrentSession`, route resolution, and connection-status loading.
  - Existing Tauri command registration path and desktop test harness runtime injection hooks.
- Outputs:
  - App-local `PersistedRemoteSession` DTOs on both the TypeScript and Rust sides.
  - Three app-local Tauri commands: `load_remote_session_cache`, `save_remote_session_cache`, `clear_remote_session_cache`.
  - Startup restore flow via `initializeDesktopConnection()`.
  - Connection-surface warning and degraded-state messaging.
- State Transitions:
  - `remote profile + valid secure cache -> restore -> revalidate -> active remote session`
  - `remote profile + expired|mismatched|corrupted secure cache -> clear cache -> /connections`
  - `remote profile + cached session + auth_required|token_expired -> clear cache + clear in-memory session -> /connections`
  - `remote profile + cached session + transport failure -> keep cached summary + degraded/read-only`
- Error Handling:
  - Secure-store read/write/delete failures become warnings and preserve memory-only login behavior.
  - Corrupted secure-cache payloads are treated as invalid and cleared locally.
  - Expired timestamps or binding mismatches are treated as invalid and cleared locally.
  - Only auth-invalid remote responses clear a locally restored session after network revalidation.
- Tech Stack Decision:
  - Reuse the existing Vue/Pinia desktop store and Tauri Rust host; add `keyring` for OS-backed secure storage.
- Visual Framework Impact:
  - Yes. The connection surface now shows explicit secure-storage warnings and degraded read-only messaging for restored sessions.
- Human Approval Points:
  - None.
- Reused Components:
  - Existing desktop connection profile storage and route resolution helpers
  - Existing `HubClientAuthError` / `HubClientTransportError` semantics
  - Existing remote auth `getCurrentSession()` revalidation path
- New Abstractions:
  - `PersistedRemoteSession`
  - `RemoteSessionCacheService`
  - `createTauriRemoteSessionCacheRuntime()`
  - `initializeDesktopConnection()`
- Trade-offs:
  - Keeping the secure cache app-local avoids widening contracts, but duplicates a small session DTO between Rust and TypeScript.
  - Preserving cached session summaries during transport failures improves resilience, but requires the UI to clearly expose degraded/read-only state.
  - Using injectable secure-session hooks keeps desktop tests deterministic without requiring OS keychain access.
- Test Strategy:
  - Red-first desktop tests for login persistence, restart restore, invalidation cleanup, degraded transport restore, and secure-store-unavailable fallback.
  - Rust tests for secure-cache save/load/clear, expiry cleanup, binding mismatch cleanup, and corrupted payload cleanup.
- ADR Needed:
  - No, unless secure-store policy becomes a reusable repository-wide rule beyond the desktop slice.

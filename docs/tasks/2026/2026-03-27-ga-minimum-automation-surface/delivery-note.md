## Delivery Note

- What Changed:
  - Added the GA minimum automation manager contract set, runtime lifecycle/list/detail surface, remote-hub automation routes, hub-client parity methods, and desktop automation list/create/detail UI.
- Why:
  - The governed runtime loop was already present; this slice closes the missing automation management surface needed to create, inspect, dispatch, and recover automations through tracked clients.
- User / System Impact:
  - Projects can now create automations for all four supported trigger types, inspect recent deliveries and last run state, pause/activate/archive automations, manually dispatch manual triggers, and retry failed deliveries from the tracked surfaces.
- Risks:
  - Desktop local mode still depends on an external bridge registration that is not implemented in the tracked repository.
  - The slice intentionally derives recent execution state from runtime records instead of a dedicated read model, so future scale-oriented dashboard work may need a separate follow-up design.
- Rollback Notes:
  - Revert the runtime migration and the coupled schema/client/remote-hub/desktop surface changes together; this slice spans one contract boundary and should not be partially rolled back.
- Follow-ups:
  - Decide later whether automation editing, secret rotation, richer delivery filtering, or a broader automation dashboard merit their own task packages.
  - If local desktop mode must work without test doubles, add a tracked implementation for `window.__OCTOPUS_LOCAL_HUB__`.
- Docs Updated:
  - Task package updated with implementation summary, verification record, and delivery note.
- Tests Included:
  - Runtime unit/contract tests, remote-hub auth and HTTP surface tests, schema-ts tests, hub-client parity tests, desktop happy-path/error-path tests, and desktop typecheck.
- ADR Updated:
  - None.
- Temporary Workarounds:
  - Desktop local transport continues to be exercised via mocked transport in tests until a tracked bridge implementation exists.

# Design Note

- Problem:
  - The repository now has a stable doc/schema-only Model Center foundation, but it has not yet frozen where those contracts will first live in persistence, how runtime may consume `ModelSelectionDecision`, or whether any read-only transport surface is needed.
- Goal:
  - Define the smallest follow-on consumer boundary for model governance without authorizing implementation or expanding into provider or capability-runtime redesign.
- Acceptance Criteria:
  - `crates/governance` and `crates/runtime` ownership is explicit.
  - The minimum read-only transport surface decision is explicit.
  - Deferred areas remain clearly out of scope for this slice.
- Non-functional Constraints:
  - Keep the work design-only and non-invasive.
  - Reuse the existing five governance contracts as the only contract baseline unless a later approved implementation slice proves otherwise.
  - Preserve ADR 0007 and the current capability-governance boundaries.
- MVP Boundary:
  - `crates/governance` may become the persistence truth for provider, catalog, profile, and tenant-policy records.
  - `crates/runtime` may consume and persist `ModelSelectionDecision` only as a per-run decision record, not as catalog truth.
  - `packages/hub-client` may later expose transport-neutral read models only if a later implementation slice proves they are necessary.
  - `apps/remote-hub` may later assemble read-only HTTP surfaces only if the later slice remains bounded to existing contracts.
- Layer Placement:
  - Persistence truth belongs in Rust crates, not in `apps/` or `packages/`.
  - Cross-language DTO truth remains in `schemas/`.
  - Read-only transport assembly belongs in `apps/remote-hub`, while `apps/desktop` remains a consumer rather than a truth owner.
- Module Boundaries:
  - `crates/governance` owns durable model-provider, model-catalog, model-profile, and tenant-policy storage semantics.
  - `crates/runtime` owns run-scoped recording and consumption of `ModelSelectionDecision`.
  - `packages/hub-client` owns transport-neutral client accessors only after an implementation slice defines them.
  - `apps/remote-hub` owns route assembly only; it does not own model-governance business rules.
  - `apps/desktop` may later consume read-only data, but it does not own persistence or policy truth.
- Inputs:
  - ADR 0007.
  - `2026-03-30-post-ga-model-center-foundation`.
  - Existing `schemas/governance` model-governance contracts.
  - Existing GA runtime / governance / transport boundaries from PRD, SAD, and the GA blueprint.
- Outputs:
  - This design-only task package.
  - A frozen ownership split for the first consumer slice.
  - A later-implementation validation strategy.
- Error Handling:
  - If future read-only transport needs new summary/detail DTOs, stop and create a bounded contract-change step in the later implementation slice rather than inventing them here.
  - If persistence ownership appears to require provider-adapter or capability-runtime redesign, stop and split the work into a different task package.
- Tech Stack Decision:
  - No new tech stack. Later implementation must stay within the existing Rust crates, SQLite, JSON Schema, and `packages/schema-ts` consumer path.
- Visual Framework Impact:
  - None.
- Human Approval Points:
  - Promoting this design package into implementation.
- Reused Components:
  - ADR 0007.
  - Existing `schemas/governance` contract set.
  - Existing crate/package boundary rules from the owner docs.
- New Abstractions:
  - None in this package. This slice only freezes ownership and consumption boundaries.
- Trade-offs:
  - Delaying implementation keeps the post-GA backlog bounded and avoids mixing persistence, transport, provider connectivity, and capability-runtime changes into one oversized slice.
- Test Strategy:
  - No code-level tests in this package.
  - Later implementation must define persistence, contract, and transport verification before coding starts.
- ADR Needed:
  - No. ADR 0007 already holds the durable boundary and terminology decision for this phase.

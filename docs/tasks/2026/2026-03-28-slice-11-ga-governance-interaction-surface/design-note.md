## Design Note

- Problem:
  - The tracked runtime already persisted approvals, inbox items, notifications, and knowledge candidates, but the minimum GA surfaces still lacked one formal governance interaction path for approval detail, inline resolution, and approval-driven knowledge promotion.
- Goal:
  - Add the minimum GA governance interaction surface so approval records remain the shared truth for both execution and knowledge promotion, and desktop / remote-hub consumers can operate on that truth without inventing new surface-local state machines.
- Acceptance Criteria:
  - Execution approvals keep their current resume / run-completion behavior.
  - Knowledge promotion enters an explicit `ApprovalRequest` flow, with one open pending approval per candidate.
  - Approval detail, request-promotion, and resolve behavior are available through the same `HubClient` shape in local and remote modes.
  - Desktop workspace / run views can inspect and act on governance items inline while respecting read-only mode.
- Non-functional Constraints:
  - No parallel contract truth outside `schemas/`.
  - No new standalone governance route tree or notification center in desktop.
  - No tenant / RBAC / IdP expansion, vector retrieval, or Org Graph work.
  - Contract behavior must stay testable in Rust and TypeScript.
- MVP Boundary:
  - Reuse the existing `WorkspaceView.vue` and `RunView.vue` instead of adding new governance routes.
  - `Notification` remains read-only; actionability lives in approval detail and inbox records.
  - `promoteKnowledge(...)` remains available as an internal runtime primitive but is not the primary GA surface path.
- Layer Placement:
  - Shared approval / inbox / notification contract changes live in `schemas/governance` and `schemas/observe`.
  - TypeScript validation and parsing live in `packages/schema-ts`.
  - Shared local / remote transport operations live in `packages/hub-client`.
  - HTTP assembly lives in `apps/remote-hub`.
  - Desktop orchestration and inline governance actions live in `apps/desktop`.
- Module Boundaries:
  - `crates/runtime` owns approval creation, promotion idempotency, approval resolution branching, and promotion side effects.
  - `packages/hub-client` exposes transport-neutral approval detail and promotion-request methods.
  - `apps/remote-hub` only maps authenticated HTTP routes to runtime operations.
  - `apps/desktop` only renders governance state and invokes the shared client.
- Inputs:
  - Existing execution approval flow, inbox / notification persistence, knowledge candidate / asset persistence, and remote-hub auth / membership enforcement.
  - Existing desktop workspace / run store and read-only connection model.
- Outputs:
  - `ApprovalRequest`, `InboxItem`, and `Notification` records with stable `target_ref`.
  - `RequestKnowledgePromotionCommand`.
  - Runtime `request_knowledge_promotion(...)` and approval resolution branching by approval type.
  - Shared `HubClient` methods for approval detail and request-promotion.
  - Remote-hub approval detail / request-promotion routes and desktop inline governance actions.
- State Transitions:
  - `execution` approvals continue to gate Run resume / completion.
  - `knowledge_promotion` approvals transition `KnowledgeCandidate` from `candidate` to `verified_shared` only on approval.
  - `reject`, `cancel`, and `expire` close the promotion approval but do not create `KnowledgeAsset` or mutate Run state.
- Error Handling:
  - Duplicate pending promotion approvals reuse the existing open record.
  - Invalid approval target or approval type returns explicit runtime errors and maps to structured HTTP 400 responses.
  - Desktop surfaces preserve the shared banner / read-only behavior instead of adding slice-local error infrastructure.
- Tech Stack Decision:
  - Keep the current stack unchanged: Rust runtime + `axum` remote-hub + Vue 3 / Pinia desktop + shared TypeScript client packages.
- Visual Framework Impact:
  - Reuse the minimum GA visual grammar and extend only the existing workspace / run panels needed for governance actions.
- Human Approval Points:
  - Execution approval remains unchanged.
  - Knowledge promotion becomes the new human-governed transition in this slice.
- Reused Components:
  - Existing `resolve_approval(...)` runtime entrypoint and execution approval semantics.
  - Existing inbox / notification persistence and remote-hub auth helpers.
  - Existing hub-client parse-and-transport pattern and desktop store/view structure.
- New Abstractions:
  - `RequestKnowledgePromotionCommand`.
  - `target_ref` as the stable approval-centric surface pointer.
  - Approval detail fetch in the shared client boundary.
- Trade-offs:
  - The slice intentionally keeps governance interactions inside existing workspace / run views rather than adding a richer governance center.
  - Promotion approval stays approval-centric and does not invent a second knowledge-governance status model.
  - Direct promotion remains as an internal primitive to avoid destabilizing older runtime composition paths while GA surfaces converge on the approval-driven flow.
- Test Strategy:
  - Write failing runtime tests first for idempotent promotion requests, approval branching, and execution-approval regression.
  - Write failing remote-hub route tests first for auth-aware approval detail and request-promotion flows.
  - Write failing schema/client tests first for local / remote transport parity and new contract shapes.
  - Write failing desktop tests first for inline governance actions, promotion requests, and read-only disabling.
- ADR Needed:
  - Not by default. Add one only if approval ownership or knowledge-promotion ownership becomes a durable repository rule beyond this slice.

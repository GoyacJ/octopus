# Repo Structure Guidelines

## 1. 目的

本文件定义 Octopus 仓库的目录职责、模块边界与放置规则。  
目标是保证 AI 与人工开发都能在同一结构约束下工作，避免共享逻辑漂移、契约分散、实现越层。

---

## 2. 总体原则

### 2.1 先定义职责，再决定放置位置

任何新文件、新模块、新对象都必须先回答：

- 它是应用装配，还是共享逻辑
- 它是领域对象，还是协议对象
- 它是运行时核心，还是界面层封装
- 它是跨语言共享契约，还是单语言内部实现

只有明确职责，才能决定目录归属。

### 2.2 共享逻辑不得塞进应用层

`apps/` 用于装配最终应用，不应用于承载可复用共享核心逻辑。

### 2.3 契约优先于实现

跨语言共享对象、关键命令、事件、DTO、状态机相关契约，优先放在 `schemas/`。

### 2.4 单一事实源

同一类对象不允许在多个目录中平行定义多个“真相版本”。

---

## 3. 顶层目录职责

## 3.1 `apps/`

用于放置最终应用或可运行装配层。

适合放：

- Desktop App
- Hub App
- Web App
- 管理台 / 控制台
- CLI 入口
- 仅服务于某个应用的组合代码

不适合放：

- 跨应用共享业务逻辑
- 跨端共享领域模型
- 共享 schema
- 通用基础设施抽象
- 核心运行时真相

### 规则

- `apps/` 只做装配，不做共享核心逻辑事实源
- 如逻辑未来需要复用，应提取到 `crates/` 或 `packages/`
- `apps/` 中的局部 UI 逻辑可以存在，但不应承载平台核心对象定义

---

## 3.2 `crates/`

用于放置 Rust 侧共享库与运行时核心能力。

适合放：

- Run / Task / Automation 核心逻辑
- Domain 模型与领域服务
- Policy / Budget / Approval 核心逻辑
- Artifact / Audit / Trace 核心逻辑
- MCP Gateway Rust 适配逻辑
- 基础设施实现库
- CLI / Hub / Service 共享库

不适合放：

- 前端专用 DTO 显示模型
- 只服务于某个单一 app 的页面代码
- 临时实验性代码
- 未沉淀稳定边界的杂糅逻辑

### 规则

- `crates/` 是 Rust 共享实现的主阵地
- 运行时核心能力优先放在 `crates/`
- 核心领域对象应保持与基础设施解耦
- crate 之间依赖必须符合分层与边界规则

---

## 3.3 `packages/`

用于放置 TypeScript / 前端侧共享包。

适合放：

- UI 组件库
- 前端共享 hooks / composables
- 前端 API client
- 前端 schema 消费层
- 前端共享状态模型
- 前端跨 app 通用工具

不适合放：

- 运行时领域真相
- 后端核心业务逻辑
- 只服务于单个页面的局部代码
- 与 `schemas/` 重复定义协议对象

### 规则

- `packages/` 是前端共享层，不是前端版“万能目录”
- 前端共享对象优先消费 `schemas/` 产物
- 不得在 `packages/` 里偷偷定义与 `schemas/` 冲突的共享契约

---

## 3.4 `schemas/`

用于放置跨语言共享契约事实源。

适合放：

- Commands
- Queries
- Events
- DTOs
- 核心枚举
- 状态机契约
- 需要跨 Rust / TypeScript 共识的结构

不适合放：

- 单语言内部 helper
- UI 专用展示模型
- 数据库 ORM 细节
- 基础设施实现逻辑
- 页面状态临时对象

### 规则

- `schemas/` 是跨语言共享契约的唯一事实源
- 共享契约变更必须先改 `schemas/`
- `crates/` 与 `packages/` 消费 `schemas/`，不得反向定义替代版本
- 若对象仅在单语言内部使用，不应强行放入 `schemas/`

---

## 4. 推荐子目录职责

以下为建议性约束，具体命名可按仓库演进调整。

## 4.1 `crates/` 推荐拆分

可按以下类型拆分：

- `runtime-*`：运行时编排与状态推进
- `domain-*`：核心领域对象与规则
- `policy-*`：授权、预算、审批
- `artifact-*`：产物、审计、追踪
- `knowledge-*`：知识候选、共享知识
- `mcp-*`：MCP 适配与网关
- `infra-*`：数据库、缓存、消息等基础设施
- `app-*`：为具体服务端应用提供装配或外壳

---

## 4.2 `packages/` 推荐拆分

可按以下类型拆分：

- `ui-*`：共享组件
- `client-*`：API client、SDK
- `schema-*`：由 `schemas/` 生成或封装的消费层
- `hooks-*` / `composables-*`：共享逻辑
- `feature-*`：跨多个 app 复用的前端特性模块

---

## 4.3 `apps/` 推荐拆分

可按以下应用面拆分：

- `desktop`
- `remote-hub`
- `web-console`
- `admin`
- `cli`

---

## 5. 目录归属决策规则

新增对象时，按以下顺序判断：

### 5.1 是否是跨语言共享契约

如果是，则优先进入 `schemas/`

### 5.2 是否是 Rust 侧共享运行时逻辑

如果是，则优先进入 `crates/`

### 5.3 是否是前端侧共享逻辑

如果是，则优先进入 `packages/`

### 5.4 是否只是某个最终应用的装配代码

如果是，则进入 `apps/`

### 5.5 是否只是局部页面、局部入口的私有逻辑

如果只服务单个应用单个场景，可保留在该 app 内，但一旦出现复用倾向，应提取

---

## 6. 严格禁止事项

- 在 `apps/` 中定义跨应用共享领域真相
- 在 `packages/` 中重定义 `schemas/` 中已有共享契约
- 在 `crates/` 中把基础设施细节渗透进领域核心
- 在多个目录中重复定义同一 command / event / DTO
- 用“临时方便”为理由长期把共享逻辑堆在 app 目录
- 跨层穿透 import / dependency
- 以 util / manager / common 等模糊目录承载核心真相

---

## 7. 与分层规则的关系

目录结构不等于架构分层，但目录必须有助于分层落地。

### 推荐理解

- `apps/` 更接近 Entry / Composition
- `crates/` 同时承载 Application / Domain / Infrastructure 的 Rust 实现，但必须再做内部层次划分
- `packages/` 承载前端共享消费层与 UI 层能力
- `schemas/` 承载跨语言协议事实源

因此，即使在同一顶层目录中，仍需继续遵守分层边界，不得因为“在同一个 crate / package”里就放松约束。

---

## 8. 与 AI 开发的配合要求

AI 在新增文件前必须先说明：

- 为什么放在这个目录
- 是否可能应该进入 `schemas/`
- 是否未来会被多个 app / package / crate 复用
- 是否会与已有对象重复
- 是否会破坏依赖方向

如无法回答上述问题，不得直接创建新模块。

---

## 9. 一句话总纲

仓库目录不是“文件摆放位置问题”，而是平台边界与真相归属问题。  
任何共享逻辑、共享契约、核心运行时对象都必须放在能体现其职责与依赖方向的位置上。

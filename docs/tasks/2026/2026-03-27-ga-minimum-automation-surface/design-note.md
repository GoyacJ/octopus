## Design Note

- Problem:
  - The repository already proves governed Automation execution, four trigger types, delivery dedupe/retry, real MCP transport, and remote-hub auth, but it still lacks any formal automation-management surface that consumes those existing runtime facts.
- Goal:
  - Add the minimum GA automation-management surface so Automation, Trigger, and recent delivery/run state can be managed through shared contracts, shared client transport, remote-hub routes, and desktop pages without changing the governed runtime main loop.
- Acceptance Criteria:
  - Project-scoped automation list/detail/create flows exist in both local and remote modes through the same `HubClient` shape.
  - Automation lifecycle exposes `active`, `paused`, and `archived` only, with explicit transition validation.
  - Manual dispatch is restricted to `manual_event` triggers and delivery retry is restricted to failed deliveries.
  - Recent execution state is shown by composing existing `TriggerDelivery` and `Run` truth.
- Non-functional Constraints:
  - No parallel contract truth outside `schemas/`.
  - No second execution shell or second execution-state store.
  - No full admin surface, no deep automation editor, and no expanded trigger model.
  - Contract behavior must stay testable in Rust and TypeScript.
- MVP Boundary:
  - One automation list view, one create flow, and one detail surface.
  - One trigger per automation remains the only supported model.
  - Lifecycle stops at `active | paused | archived`; `draft` and `suspended` remain reserved target-state semantics.
  - No automation-specific live streaming surface beyond the existing transport/event infrastructure.
- Layer Placement:
  - Shared commands, DTOs, and response shapes live in `schemas/runtime`.
  - TypeScript validation and parsing live in `packages/schema-ts`.
  - Shared local/remote transport operations live in `packages/hub-client`.
  - HTTP assembly lives in `apps/remote-hub`.
  - Desktop routing, store orchestration, and view composition live in `apps/desktop`.
- Module Boundaries:
  - `crates/runtime` owns lifecycle rules, automation reads, and delivery/run composition.
  - `packages/hub-client` exposes one transport-neutral automation-management interface and hides local/remote transport differences.
  - `apps/remote-hub` only maps HTTP routes to runtime operations and schema-owned payloads.
  - `apps/desktop` only renders automation-management state and invokes the shared client.
- Inputs:
  - Existing `CreateAutomationInput`, `CreateTriggerInput`, `DispatchManualEventInput`, and `retry_trigger_delivery(...)` runtime entrypoints.
  - Existing `AutomationRecord`, `TriggerRecord`, `TriggerDeliveryRecord`, and `RunRecord` persistence.
  - Existing auth-aware remote transport boundary.
- Outputs:
  - Shared automation create/list/detail/lifecycle/delivery DTOs.
  - Runtime automation list/detail/lifecycle methods.
  - Shared `HubClient` automation methods.
  - Remote-hub automation routes and desktop automation pages.
- State Transitions:
  - Automation lifecycle subset for this slice: `active -> paused`, `paused -> active`, `active -> archived`, `paused -> archived`.
  - Newly created automations become `active` immediately in this slice.
  - Delivery retry remains limited to existing failed-delivery recovery semantics.
- Error Handling:
  - Invalid lifecycle transitions return explicit runtime errors and map to structured HTTP 400 responses.
  - Manual dispatch against non-`manual_event` triggers is rejected explicitly.
  - Delivery retry against non-failed deliveries is rejected explicitly.
  - Desktop surfaces show transport/runtime errors directly and preserve auth-driven read-only behavior.
- Tech Stack Decision:
  - Keep the current stack unchanged: Rust runtime + `axum` remote-hub + Vue 3/Pinia desktop + shared TypeScript client packages.
- Visual Framework Impact:
  - Reuse the existing minimum GA visual grammar and extend navigation/pages only where automation management requires it.
- Human Approval Points:
  - None.
- Reused Components:
  - Existing runtime automation projection model.
  - Existing trigger-specific ingress and delivery retry behavior.
  - Existing `schema-ts`, `hub-client`, remote-hub auth, and desktop store/view patterns.
- New Abstractions:
  - `AutomationSummary` and `AutomationDetail` shared surface payloads.
  - Thin runtime automation lifecycle/read APIs.
  - Desktop automation list/detail routes and store actions.
- Trade-offs:
  - The slice intentionally implements the minimum lifecycle subset required for surface management rather than the full SAD target-state automation state machine.
  - Recent execution is derived at read time to avoid introducing a second automation status store.
  - Webhook secret handling stays minimal and one-time reveal only.
- Test Strategy:
  - Write failing runtime tests first for lifecycle, list/detail, and delivery/run derivation.
  - Write failing remote-hub route tests first for auth-aware automation create/list/detail/lifecycle flows.
  - Write failing schema/client tests first for local/remote transport parity.
  - Write failing desktop tests first for create/list/detail and read-only/error behavior.
- ADR Needed:
  - Not by default. Add one only if lifecycle ownership or recent-execution derivation becomes a durable repository rule beyond this slice.

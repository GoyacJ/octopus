# Code Review Checklist

## 1. 目的

本清单用于人工评审与 AI 自检。  
目标是将仓库级工程规范、阶段门规范与实施蓝图转化为可执行的评审项，降低以下风险：

- 范围漂移
- 跨层实现
- 契约失配
- 巨型文件与巨型类
- 测试不足
- 临时方案伪装成最终方案
- 文档、实现、测试不一致

---

## 2. 使用方式

建议在以下场景使用：

- AI 完成一次任务后的自检
- 人工 Code Review
- PR 合并前最终检查
- 重构与公共接口变更评审
- schema / ADR 相关改动评审

建议按“先大后小”的顺序评审：

1. 范围与目标
2. 架构与分层
3. 契约与目录归属
4. 实现质量
5. 测试与验证
6. 文档与交付说明
7. Git 变更卫生

---

## 3. 范围与目标检查

- 本次改动的目标是否清晰
- Scope 与 Out of Scope 是否明确
- Acceptance Criteria 是否明确
- 非功能约束是否明确且与改动匹配
- MVP slice 是否足够小且与当前任务一致
- Human Approval Points 是否明确
- 是否存在顺手扩大范围
- 是否把一个局部任务做成了隐含重构
- 是否有非必要的大面积重排
- 是否有明显“未来需求预埋”但当前无真实必要
- 是否与本次任务目标保持一致

### 重点警示

以下情况通常应要求回退：

- 没有明确目标就开始改动
- 验收标准缺失却已进入实现
- 关键 NFR 未说明却已实现关键路径
- 改动范围明显超出任务定义
- 把“顺手优化”做成大范围结构变更
- 没有说明为何要引入额外抽象

---

## 4. 架构与分层检查

- 代码是否位于正确层级
- 入口层是否只做参数接收、校验、协议适配、响应映射
- 应用层是否只做用例编排与流程控制
- 领域层是否只承载核心规则与不变量
- 基础设施层是否避免业务决策
- 是否存在跨层穿透调用
- 是否存在反向依赖
- 是否存在 UI / 协议逻辑渗透进运行时核心
- 是否存在基础设施细节渗透进领域核心

### 重点警示

以下情况通常不应通过：

- 控制器直接访问数据库或第三方服务
- 入口层编排大量核心逻辑
- 领域对象依赖具体数据库 / 网络实现
- 在基础设施层偷偷写规则判断
- 以“方便”为理由绕过分层

---

## 5. 目录与模块归属检查

- 新增文件放置位置是否合理
- 是否误把共享逻辑放进 `apps/`
- 是否误把共享契约放在 `packages/` 或 `crates/` 而非 `schemas/`
- 是否存在同一类对象的重复真相
- 是否存在 util / manager / common 等模糊目录承载核心逻辑
- 模块命名是否体现职责
- 模块边界是否清晰
- 是否为了“先写完”而把多个职责塞进同一模块

### 重点警示

以下情况通常不应通过：

- 在 app 目录中定义共享领域对象
- 在多个位置重复定义 command / event / DTO
- 模块命名模糊且无法判断职责
- 目录归属理由说不清楚

---

## 6. Schema / 契约检查

- 是否涉及跨语言共享对象
- 若涉及，是否已先更新 `schemas/`
- 是否新增或修改了 command / query / event / DTO / 状态枚举
- 是否补充兼容性说明
- 是否说明受影响消费者
- 是否说明迁移要求
- 是否存在 Rust 与 TypeScript 各写一套近似但不一致的对象
- 是否补充了契约测试或映射测试

### 重点警示

以下情况通常不应通过：

- 修改公共接口但没有契约说明
- schema 变更没有兼容性说明
- 让 UI 模型或内部模型冒充公共契约
- 修改事件语义但未说明影响面

---

## 7. 设计与抽象检查

- 本次改动是否真的需要新增抽象
- 是否出现过度设计
- 设计模式是否有明确收益
- 是否为了未来假想需求提前抽象
- 是否引入复杂责任链、工厂族、事件体系但无必要
- 是否有更简单的实现方式
- 是否保留了清晰依赖方向
- 是否使理解路径变得更复杂

### 重点警示

以下情况通常不应通过：

- 只有一个简单分支却引入策略体系
- 简单对象创建却引入工厂族
- 没有真实变化点却建立大量抽象接口
- 以“以后可能会用到”为主要理由做复杂设计

---

## 8. 实现质量检查

- 文件大小是否可控
- 是否出现巨型文件
- 函数长度是否可控
- 嵌套层级是否合理
- 是否存在 God Object / God Service
- 是否存在大量隐藏副作用
- 是否存在魔法数字、魔法字符串
- 命名是否清晰稳定
- 是否使用了显式错误处理
- 是否有无关代码改动
- 是否有不必要的大量风格重排

### 文件与函数参考阈值

- 单文件建议 200–400 行
- 超过 500 行应重点审查
- 超过 800 行原则上不接受
- 普通函数建议 20–50 行
- 超过 80 行应重点审查
- 嵌套通常不超过 3 层

### 重点警示

以下情况通常不应通过：

- 一个类同时承担编排、规则、IO、映射、校验
- 一个文件同时塞 controller / service / repository / DTO / helper
- 用注释解释一大段无法维护的复杂实现
- 显然需要拆分却没有拆

---

## 9. 错误处理与安全检查

- 是否区分用户错误、业务错误、系统错误、外部依赖错误
- 是否按“分类 -> 定位 -> 处置”处理关键错误路径
- 是否保留足够上下文
- 是否在边界做错误归一化
- 是否有错误码映射或结构化响应
- 是否存在吞异常
- 是否打印错误后假装成功
- 是否存在敏感信息泄露风险
- 是否对外部输入做基础校验
- 是否存在命令、查询、路径拼接风险
- 是否在日志中输出敏感数据

### 重点警示

以下情况通常不应通过：

- `catch` 后无处理
- 用布尔值替代详细错误
- 只记录报错但没有处置策略
- 错误链断裂导致无法追踪
- 把内部实现细节直接暴露给外部调用方

---

## 10. 测试与验证检查

- 是否补了测试
- 是否覆盖核心路径
- 是否覆盖失败路径
- 是否覆盖边界条件
- 是否覆盖回归风险点
- 是否有最小集成验证
- 是否验证了契约行为
- 测试是否稳定、可复现
- 是否过度依赖 mock
- 是否仅验证调用次数而不验证结果
- 是否只有 happy path

### 重点警示

以下情况通常不应通过：

- 新增核心逻辑没有测试
- 改了公共行为却没有回归测试
- 失败路径完全未测
- 使用脆弱快照掩盖行为验证不足

---

## 11. 注释、文档与交付说明检查

- 是否补了必要注释
- 注释是否解释“为什么”而不是翻译代码
- 是否有过期或误导性注释
- 产品语义变化是否更新 PRD
- 任务级设计与验收是否进入 task package
- 共享契约变化是否遵守 schema-first
- UI / Surface 任务是否正确处理 `VISUAL_FRAMEWORK.md`
- 是否更新相关文档
- 是否补充 Delivery Note
- 是否说明风险点
- 是否标注临时方案
- 是否说明后续收敛项
- 代码、测试、文档是否一致
- 是否需要新增或更新 ADR

### 重点警示

以下情况通常不应通过：

- 临时方案未标注
- 代码已变更但文档仍描述旧行为
- 把任务级局部设计错误写进 PRD
- 把非 UI 工作错误卡在视觉框架前置门上
- 注释与实现不一致
- 明显架构结论没有 ADR 留痕

---

## 12. Git 变更卫生检查

- 本次 change set 是否围绕单一目标或单一 MVP slice
- diff 是否与 Task Definition 和 Out of Scope 保持一致
- 是否混入无关格式化、重命名、搬移或“顺手修”
- 是否保留了可追溯的验证证据
- 是否提供了可理解的 rollback notes
- 是否对生成文件、锁文件或大批量自动变更给出理由

### 重点警示

以下情况通常不应通过：

- 把行为变更和大面积格式化混在同一次改动里
- 没有 fresh verification evidence 就宣称通过
- 无法说明为什么这些文件必须一起改
- 非平凡改动没有回滚说明

---

## 13. PR 最终放行问题

在允许合并前，至少确认以下问题都能回答为“是”：

- 目标是否清楚且范围受控
- 验收标准、NFR 与 MVP 边界是否清楚
- 分层是否正确
- 目录归属是否正确
- 共享契约是否正确处理
- 实现是否可维护
- 测试是否足够
- 风险是否透明
- 临时方案是否透明
- Git 变更卫生是否达标
- 文档与代码是否一致
- 本次改动是否具备可持续演进性

---

## 14. 一句话总纲

Code Review 不是确认“代码能跑”，而是确认这次改动在目标、结构、契约、实现、测试与交付上都足够稳，值得进入仓库长期维护。

## 1. 产品概述

### 1.1 产品定位

**Octopus** —— 让每个人都能拥有并管理一支专属 AI 工作团队。

用户可以创建多个具有独立角色、性格、记忆和技能的 AI Agent，将其组织成团队，像管理真实员工一样下发任务、审批关键操作、查看执行结果，也可以召集多个 Agent 进行圆桌会议、头脑风暴，让不同专业视角的 Agent 共同讨论、碰撞观点、收敛结论。系统支持私有化部署，客户端连接 Hub 执行层运行，数据和计算完全由用户掌控。

### 1.2 核心价值主张

| 用户痛点 | Octopus 的解法 |
|---------|------------|
| 单一 AI 助手无法处理复杂多角色任务 | 多 Agent 协作，每个 Agent 专职一个角色 |
| AI 没有记忆，每次对话从零开始 | 每个 Agent 拥有属于自己的长期记忆，随任务积累持续成长 |
| 无法信任 AI 自主执行高风险操作 | 用户审批节点 + 能力白名单权限控制 |
| 企业/个人数据不愿上传公有云 | 私有化部署 + 本地 Hub，数据不出本地 |
| 配置 AI 门槛高，需要懂提示词工程 | 预制模板 + 可视化配置，30 分钟上手 |
| 需要多方视角讨论但缺少专家资源 | 圆桌会议模式，召集多个专业 Agent 共同探讨 |

### 1.3 竞品差异化定位

| 对比对象 | Octopus 的差异 |
|---------|------------|
| ChatGPT / Claude | 不是单点对话，是可编排的 Agent 团队，有持久记忆和角色分工 |
| Dify / Coze | 不是 Workflow 构建器，是以"人"为中心的 Agent 管理系统，面向非技术用户 |
| AutoGen / CrewAI | 有完整产品 UI，无需编程，支持私有化部署和多端客户端 |
| Notion AI / Copilot | 不是嵌入式辅助，而是完整的 Agent 团队操作系统，支持多 Agent 圆桌讨论 |

### 1.4 核心假设

Octopus 的产品定义围绕以下四个核心假设展开：

- **假设 1**：单 Agent 个性化 + 记忆能力，能提升用户对 AI 的依赖度
- **假设 2**：多 Agent 协作完成复杂任务，比单 Agent 有明显体验优势
- **假设 3**：用户愿意像管理团队一样管理 Agent（管理体验）
- **假设 4**：多 Agent 圆桌讨论能帮助用户从多专业视角快速获得高质量洞察

## 2. 目标用户与核心场景

### 2.1 用户分层

| 层级 | 用户类型 | 典型规模 | 核心诉求 |
|-----|---------|---------|---------|
| **P0** | 个人高级用户 / One Person Company | 1 人 | 用 Agent 团队代替外包，低成本完成多职能工作 |
| **P1** | 小型技术 / 产品团队 | 5–30 人 | 多人共享 Agent 团队，协作完成项目任务 |
| **P2** | 中大型企业 / 组织机构 | 30 人以上 | 私有化部署，精细权限管理，跨部门 Agent 协作 |

### 2.2 核心使用场景

**场景 1：One Person Company 全职能助手**
> 独立创业者配置：市场推广 Agent + 财务管理 Agent + 客服 Agent + 技术研发 Agent，下达"本月完成产品上线并启动推广"，各 Agent 协作分工执行。

**场景 2：产品团队需求分析**
> PM 配置：市场调研 Agent + 竞品分析 Agent + 技术可行性评估 Agent，输入一段产品想法，团队自动产出完整需求分析报告。

**场景 3：企业事件响应团队**
> 运营负责人配置：舆情监测 Agent + 危机处理 Agent + 媒体沟通 Agent，7×24 监控，在触发条件时自动启动响应流程，关键决策推送用户审批。

**场景 4：项目管理自动化**
> 团队负责人配置：任务分配 Agent + 进度跟踪 Agent + 风险评估 Agent，每日自动汇报项目状态，异常时主动预警。

**场景 5：内容营销团队**
> 内容负责人配置：SEO 分析 Agent + 内容创作 Agent + 数据分析 Agent，每周自动产出内容计划并执行。

**场景 6：圆桌讨论 — 产品需求评审**
> 用户创建产品设计师 Agent、研发工程师 Agent、测试工程师 Agent，召集三者进行需求圆桌，用户输入需求描述，三个 Agent 从各自专业视角讨论可行性、边界条件、测试要点，最终收敛出评审意见。

**场景 7：圆桌讨论 — 多专家头脑风暴**
> 用户创建多个行业专家 Agent（如 3 个不同风格的产品设计师），针对同一设计问题让他们相互碰撞，激发不同方向的创意，最终由用户选取最优方案。

**场景 8：圆桌讨论 — 投资决策讨论**
> 用户创建多个炒股专家 Agent（价值投资派、技术分析派、量化派），针对某只股票或市场走势进行多空辩论，从不同流派角度分析，用户综合判断。

## 3. 系统架构与部署模型

### 3.1 核心架构概念

```
┌──────────────────────────────────────────────────────────────┐
│                         CLIENT 层                             │
│                                                              │
│   Desktop App           Mobile App          Web Browser      │
│  (Client + 内嵌Hub)     (Client Only)       (Client Only)    │
└──────────┬──────────────────────┬─────────────────┬──────────┘
           │   Hub Protocol       │                 │
           │   (HTTPS + Auth)     │                 │
┌──────────▼──────────────────────▼─────────────────▼──────────┐
│                           HUB 层                              │
│                                                              │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────────┐  │
│  │  Agent 运行时 │  │  任务调度引擎  │  │     存储层        │  │
│  │  (执行核心)   │  │ (Leader 协调) │  │  向量DB + 关系DB  │  │
│  └──────────────┘  └───────────────┘  └──────────────────┘  │
│                                                              │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────────┐  │
│  │  讨论引擎     │  │   MCP 网关    │  │    多租户管理      │  │
│  │ (圆桌/头脑风暴)│  │  (工具接入)   │  │                  │  │
│  └──────────────┘  └───────────────┘  └──────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

**核心原则**：
- **Hub = 执行层**：所有 Agent 运行、任务调度、讨论引擎、数据存储、权限管理均在 Hub
- **Client = 交互层**：纯 UI 渲染，无状态，不存储业务数据
- **Desktop 特殊**：桌面客户端内嵌本地 Hub，默认即可离线使用

### 3.2 部署模式详述

#### 模式 A：桌面端本地模式（默认）

- 下载桌面应用，内嵌本地 Hub，无需网络，完全离线可用
- 数据存储在本地（SQLite + 本地向量 DB）
- 适合个人用户、隐私敏感场景
- 无需注册账号即可使用

#### 模式 B：桌面端连接远程 Hub

- 桌面 Client 连接到企业内网或云端部署的 Hub
- 需要登录认证（JWT / OAuth2），认证成功后 Client 获取操作权限
- 执行层在远程 Hub，本地仅渲染 UI
- 本地 Hub 与远程 Hub 可同时存在，独立切换

#### 模式 C：移动端连接远程 Hub

- 移动端无本地 Hub，必须连接远程 Hub 使用
- 认证流程同模式 B
- 支持离线查看历史记录（本地缓存），但不能执行新任务

#### 模式 D：私有化部署 Hub

- 企业在自有服务器 / 内网部署完整 Hub 服务
- 支持 Docker / K8s 部署包
- 多租户隔离，每个租户独立的 Agent / Team / 数据空间
- 管理员通过 Hub 控制台统一管理

### 3.3 多 Hub 管理

- 单个 Client 支持注册并连接**多个 Hub**
- 每个 Hub 用自定义名称标识（如"公司内网 Hub"、"个人本地 Hub"）
- Client UI 顶部支持快速切换当前活跃 Hub
- 不同 Hub 的数据完全隔离，不跨 Hub 共享 Agent 或记忆
- Hub 连接状态实时展示（在线 / 离线 / 认证失效）

### 3.4 Hub 认证流程

```
Client 添加远程 Hub
        │
        ▼
输入 Hub 地址（URL / IP:Port，支持扫描二维码）
        │
        ▼
Hub 握手，返回支持的认证方式
（用户名密码 / SSO / API Key）
        │
        ▼
用户完成认证 → 获取 Access Token + Refresh Token
        │
        ▼
Client 本地安全存储 Token
（macOS Keychain / Windows Credential Store / Android Keystore）
        │
        ▼
后续请求自动携带 Token，Token 过期自动刷新
```

## 4. 核心概念模型

### 4.1 实体关系总览

```
Tenant（租户）
    │
    ├── User（用户）
    │       └── 创建 / 拥有 → Agent
    │
    ├── Team（团队）
    │       ├── members: Agent[]
    │       ├── leader: Agent（可选）
    │       └── 执行 → Task
    │
    ├── DiscussionSession（讨论会话） 
    │       ├── participants: Agent[]
    │       ├── moderator: Agent（可选）
    │       └── turns: DiscussionTurn[]
    │
    └── TeamGroup（团队组）
            └── 包含多个 Team
```

### 4.2 Agent 的本质：数字化的人

Octopus 中的 Agent 不是一个"功能模块"或"API 封装"，而是对现实世界中**"人"的数字化抽象**。每个 Agent 都是一个独立的个体，天然拥有三个不可分割的维度：

**Identity（身份）**：他是谁。包括名字、外貌、角色定位、性格特质。这决定了他与人交互的方式和风格。

**Capability（能力）**：他能做什么。通过工具白名单、Skill 附加、MCP 绑定来定义其可执行的操作边界。

**Memory（记忆）**：他知道什么、经历过什么。记忆随任务和讨论积累，不断丰富。

> **关于圆桌会议中的 Agent**：Agent 在讨论中体现的是 Identity 维度——他的专业立场、思维方式、表达风格。讨论中 Agent 通常不调用工具（除非用户显式开启"工具增强讨论"），但他的历史记忆会自动注入，使他能在讨论中引用过往经验。

### 4.3 Agent 实体定义

| 属性 | 类型 | 维度 | 说明 |
|-----|------|------|------|
| `id` | string | — | 唯一标识 |
| `name` | string | Identity | Agent 名称（如 "Alex · 市场分析师"） |
| `avatar` | string | Identity | 头像 URL（自定义上传或 AI 生成） |
| `role` | string | Identity | 角色描述（自然语言，如 "负责市场竞品分析"） |
| `persona` | string[] | Identity | 性格标签（如 ["严谨", "逻辑性强", "简洁"]） |
| `system_prompt` | string | Identity | 核心 System Prompt，支持 `{{变量}}` 语法 |
| `model_profile_id` | string | Capability | 绑定的推理模型档案 ID（从租户允许的模型档案中选择） |
| `tools_whitelist` | string[] | Capability | 允许使用的工具列表（能力白名单） |
| `skill_ids` | string[] | Capability | 附加的 Skill ID 列表（运行时合并到有效配置） |
| `mcp_bindings` | string[] | Capability | 绑定的 MCP 服务 ID 列表 |
| `memory_store_id` | string | Memory | 对应的向量记忆库 ID（Agent 私有，不对外直接暴露） |
| `created_by` | string | — | 创建者用户 ID |
| `status` | enum | — | `idle` / `busy` / `error` |

### 4.4 Team 实体定义

| 属性 | 类型 | 说明 |
|-----|------|------|
| `id` | string | 唯一标识 |
| `name` | string | 团队名称 |
| `description` | string | 团队职能描述 |
| `members` | string[] | 成员 Agent ID 列表（不含 Leader） |
| `leader_id` | string | Leader Agent ID（必填；创建时自动生成或从现有 Agent 中选择） |
| `leader_source` | enum | `auto_generated`（系统创建）/ `user_selected`（选择已有 Agent） |
| `routing_rules` | Rule[] | 用户自定义路由规则 |
| `score` | number | 综合评分（基于历史任务完成情况，用于团队治理与路由决策） |
| `task_history` | string[] | 历史任务 ID 列表 |

### 4.5 Task 实体定义

| 属性 | 类型 | 说明 |
|-----|------|------|
| `id` | string | 唯一标识 |
| `input` | string | 用户原始任务描述（自然语言） |
| `status` | enum | `pending` / `planning` / `running` / `waiting_approval` / `terminated` / `completed` / `failed` |
| `mode` | enum | `single_agent`（单 Agent 执行）/ `team`（团队协作执行） |
| `plan` | Subtask[] | Leader 生成的执行计划（含阶段和依赖关系） |
| `pipeline_approval_mode` | enum | `per_stage`（每阶段成果确认）/ `auto_approve`（全链路自动放行）/ `custom`（自定义各阶段策略） |
| `result` | string | 最终汇总结果 |
| `pending_decisions` | Decision[] | 待用户确认的决策队列（Leader 汇总后统一推送） |
| `trace_log` | LogEntry[] | 完整执行日志 |
| `created_at` | timestamp | 任务创建时间 |
| `completed_at` | timestamp? | 任务完成时间 |
| `terminated_at` | timestamp? | 任务被终止时间 |
| `terminated_by` | string? | 发起终止操作的用户 ID |

### 4.6 Decision 实体定义

Agent 在执行过程中遇到需要决策的情况时，向 Leader（或直接向用户）提交 Decision 请求。

| 属性 | 类型 | 说明 |
|-----|------|------|
| `id` | string | 唯一标识 |
| `task_id` | string | 所属任务 ID |
| `source_agent_id` | string | 提出决策请求的 Agent ID |
| `type` | enum | `deliverable_review`（成果物审核）/ `clarification`（执行疑问）/ `risk_alert`（风险提示） |
| `content` | string | 决策请求内容（问题描述或成果物摘要） |
| `artifact_ref` | string? | 关联的成果物引用（如文档、代码片段） |
| `status` | enum | `pending`（待决策）/ `auto_resolved`（Leader 基于记忆自动处理）/ `user_confirmed`（用户确认）/ `user_rejected`（用户拒绝） |
| `resolution` | string? | 最终决策结果（来自 Leader 记忆或用户输入） |
| `memory_ref` | string? | 若 Leader 自动处理，指向 Leader 记忆库中沿用的历史决策条目 ID |
| `created_at` | timestamp | |

### 4.7 DiscussionSession 实体定义  

圆桌会议 / 头脑风暴的核心实体，独立于 Task 存在。

| 属性 | 类型 | 说明 |
|-----|------|------|
| `id` | string | 唯一标识 |
| `topic` | string | 讨论议题（用户输入，自然语言描述问题或需求） |
| `mode` | enum | `roundtable`（圆桌，各方平等发言） / `brainstorm`（头脑风暴，鼓励发散） / `debate`（辩论，立场对立） |
| `status` | enum | `active`（进行中） / `paused`（暂停） / `concluded`（已结束） |
| `participant_ids` | string[] | 参与讨论的 Agent ID 列表（2–8 个） |
| `moderator_id` | string? | 主持人 Agent ID（可选；可从参与者中指定，也可自动生成） |
| `turn_strategy` | enum | `sequential`（顺序轮流） / `moderated`（主持人指定） |
| `max_turns` | number | 每个 Agent 最大发言轮数（默认 10，超出后自动触发总结） |
| `context_window` | number | 发言上下文保留的最近 N 条（默认 10，更早的内容滚动摘要） |
| `conclusion` | string? | 最终结论摘要（结束时由系统合成） |
| `created_by` | string | 创建者用户 ID |
| `created_at` | timestamp | 创建时间 |
| `concluded_at` | timestamp? | 结束时间 |

**DiscussionTurn 实体**：

| 属性 | 类型 | 说明 |
|-----|------|------|
| `id` | string | 唯一标识 |
| `session_id` | string | 所属讨论会话 |
| `turn_number` | number | 全局轮次序号（从 1 开始递增） |
| `speaker_type` | enum | `agent`（Agent 发言） / `user`（用户注入） / `system`（系统提示） |
| `speaker_id` | string? | agent_id 或 user_id |
| `content` | string | 发言内容 |
| `created_at` | timestamp | 发言时间 |

### 4.8 Agent 的成长性

一个 Agent 刚被创建时，他的记忆是空白的——就像一个新入职的员工，掌握了岗位技能，但还没有在这家公司积累经验。

随着任务的不断执行，Agent 会将每次任务中的关键经验、用户偏好、领域知识、历史决策依据写入自己的记忆。下次遇到类似任务时，他会主动回忆相关经验，做出更快、更准确的判断。

这种成长是**持续的、个人的、不可被其他 Agent 共享的**——每个 Agent 的记忆库构成了他独特的"个人经验"。

在 UI 上，用户可以查看每个 Agent 积累了多少条记忆、最近在哪些任务中有所成长，就像查看一个员工的成长档案一样；必要时可进一步提供成长趋势的可视化呈现。

## 5. 功能需求详述

### 5.1 Agent 管理

#### 5.1.1 创建与配置 Agent

**用户故事**：作为用户，我希望创建一个具有独特角色和能力的 Agent，以便将其加入我的工作团队。

**基础信息配置（Identity）**：
- 名称、头像（上传或 AI 生成）
- 角色描述（自然语言输入）
- 性格标签（多选，如：严谨 / 创意 / 执行力强 / 善于沟通）

**模型配置（Capability）**：
- 从当前租户允许的 `ModelProfile` 列表中选择推理模型档案，而不是直接填写 `provider + model`
- UI 展示档案摘要：档案名称、Provider、实际模型 ID、参数预设
- 默认值来自租户级 `TenantModelPolicy`
- 高级参数由管理员在模型档案中统一固化，普通用户不直接编辑 API Key、端点或裸模型字符串

**Prompt 配置（Identity）**：
- System Prompt 富文本编辑器，支持 `{{变量名}}` 动态变量
- 内置 Prompt 模板库，一键导入常用角色 Prompt
- Prompt 版本历史，支持查看 diff 和回滚

**能力白名单（Capability · Tools）**：

配置时通过勾选方式为 Agent 授权工具，按风险等级分组展示：

| 分组 | 工具 | 风险等级 |
|-----|------|---------|
| 文件操作 | 读取文件、写入文件、搜索文件、列目录 | 🟢 读 / 🟡 写 |
| 代码执行 | 执行 Python、执行 Bash、执行 Node.js | 🟡 中 |
| 网络访问 | HTTP GET、HTTP POST、网页抓取 | 🟡 中 |
| 数据处理 | JSON 解析、CSV 处理、数据转换 | 🟢 低 |
| 系统操作 | 进程管理、环境变量读取、系统命令 | 🔴 高 |

**Skill 附加（Capability · Skills）**：
- 从 Skill 库中选择附加的能力模块（如"软件开发"、"信息调研"）
- 附加 Skill 后，Agent 自动获得 Skill 授权的工具 + Prompt 增强片段
- UI 实时预览附加 Skill 后的有效工具集合

**MCP 绑定（Capability · MCP）**：
- 从 Hub 已注册的 MCP 服务列表中选择绑定
- 可细粒度启用 / 禁用该 MCP 服务暴露的具体工具

**Agent 测试对话**：
- 配置完成后可单独与该 Agent 对话，验证性格、Prompt 效果
- 显示 Agent 实际使用的 System Prompt（Debug 模式）

**验收标准**：
- 可在 5 步以内完成一个基础 Agent 的创建
- 切换模型后，下一次对话立即生效
- Prompt 修改后有版本记录，可一键回滚

---

#### 5.1.2 Agent 记忆（Memory 维度）

**用户故事**：作为用户，我希望 Agent 能够积累自己的经验和偏好，而不是每次任务都从零开始——就像一个有经验的员工，不需要反复交代同样的背景。

**Agent 如何形成记忆**：

每次任务完成后，Agent 会自动将本次任务中的关键经验写入自己的记忆，包括：重要结论、用户偏好、领域知识、历史决策依据。这个过程由 Agent 自主完成，不需要用户干预。

**Agent 如何使用记忆**：

Agent 在执行每次任务前，会主动检索自己记忆中与当前任务相关的内容，并将其作为背景知识融入当前对话。用户不需要重复交代已经告知过的信息。

**记忆的个人性与隔离性**：

每个 Agent 的记忆完全属于他自己，不会与其他 Agent 共享。Agent A 的记忆不会影响 Agent B 的判断。不同租户之间的记忆数据完全隔离。

**记忆管理 UI**：

用户可以在 Agent 详情页查看和管理该 Agent 的记忆：
- 以列表形式展示记忆条目（时间 + 内容摘要 + 来源任务）
- 支持：手动添加记忆 / 删除错误记忆 / 全文搜索记忆
- 每条记忆可追溯到具体的来源任务
- 显示 Agent 记忆总量和最近成长情况

**验收标准**：
- 第二次执行同类任务时，Agent 能主动引用上次相关结论
- 用户可在 UI 中查到某条记忆的来源任务
- 删除记忆后，Agent 在后续任务中不再使用该条记忆

### 5.2 团队管理

#### 5.2.1 创建团队

**Leader 是团队的必要角色**，每个 Team 必须有且只有一个 Leader Agent。Leader 本质上也是一个 Agent——他有自己的 Identity、Capability 和 Memory，只不过他的职责是协调分工、分解任务、汇总决策。

创建流程中提供两种方式：

**方式 A：自动生成 Leader（推荐，默认）**
- 系统基于团队名称和职能描述，自动生成一个专职 Leader Agent
- 自动生成的 Leader 预置：协调分工 Prompt + 决策记忆能力 + 任务分解工具
- 用户可在生成后修改 Leader 的名称、头像和 Prompt
- 适合快速创建、不想手动配置 Leader 的场景

**方式 B：从现有 Agent 中指定 Leader**
- 用户从自己的 Agent 列表中选择一个已有 Agent 担任 Leader
- 该 Agent 将同时承担成员角色和管理角色
- 适合已有专属 Leader Agent 配置的用户

**创建步骤**：
1. 输入团队名称、职能描述
2. 选择 Leader 方式（自动生成 / 选择现有）
3. 从 Agent 列表中多选普通成员 Agent
4. 可选：配置路由规则（见 5.2.2）
5. 确认创建

#### 5.2.2 Leader 任务分解机制

**设计原则**：规则优先，LLM 兜底，执行前可视化确认。

Leader 之所以能够协调团队，得益于他自身积累的记忆——他记得每个成员 Agent 擅长什么、过去哪类任务交给哪个 Agent 效果最好。这些经验随任务积累不断优化，使 Leader 的分配决策越来越准确。

```
用户下发任务
      │
      ▼
Leader 检查用户定义的路由规则
      │
      ├── 规则命中 ──→ 按规则直接分配子任务
      │
      └── 规则未命中
              │
              ▼
        Leader 回忆自身记忆（历史分配经验）
        + 分析所有成员 Agent 的能力描述
        生成结构化执行计划（JSON）
              │
              ▼
        执行计划展示给用户预览
        （简单任务：自动跳过 / 复杂任务：强制确认）
              │
              ▼
        用户确认 / 修改 / 重新规划
              │
              ▼
        分发子任务，Agent 并行 / 串行执行
```

**执行计划数据结构**（Leader 输出）：

```json
{
  "task_id": "task_001",
  "analysis": "任务需要竞品调研 + 技术可行性评估，建议并行执行后汇总",
  "subtasks": [
    {
      "id": "sub_1",
      "agent_id": "agent_market",
      "description": "调研竞品 A、B、C 的定价策略和功能对比",
      "depends_on": [],
      "approval_required": false
    },
    {
      "id": "sub_2",
      "agent_id": "agent_tech",
      "description": "评估技术实现可行性，输出关键风险点",
      "depends_on": [],
      "approval_required": false
    },
    {
      "id": "sub_3",
      "agent_id": "agent_product",
      "description": "基于竞品分析和技术评估，产出产品策略建议报告",
      "depends_on": ["sub_1", "sub_2"],
      "approval_required": true
    }
  ]
}
```

**路由规则配置示例**：

```
IF   任务描述包含关键词 "代码" 或 "bug"
THEN 分配给 [工程师 Agent]

IF   任务来源标签 = "紧急"
THEN 跳过计划确认，直接执行，并通知用户

IF   任务涉及工具 = "外部 API 写操作"
THEN 强制触发用户审批节点
```

#### 5.2.3 Agent 决策反馈机制

Agent 在执行过程中遇到**疑问**或产出**成果物**时，不直接阻塞等待用户，而是通过决策反馈机制进行异步处理。团队模式和单 Agent 模式的处理路径不同。

---

**团队模式决策流程**：

```
Agent 遇到疑问 或 产出阶段成果物
        │
        ▼
Agent 将 Decision 请求提交给 Leader
（包含：类型 / 内容 / 成果物引用）
        │
        ▼
Leader 查询自身记忆（他过去如何处理过类似情况）
        │
        ├── 记忆命中：找到相似历史决策
        │       │
        │       ▼
        │   Leader 自动沿用历史决策（状态：auto_resolved）
        │   同时将本次沿用记录写入自己的记忆
        │   在 UI 侧显示："Leader 已基于历史经验自动处理，点击查看"
        │
        └── 记忆未命中：无历史决策可参考
                │
                ▼
        Leader 将该 Decision 加入待确认队列
        Leader 可批量收集多个 Agent 的 Decision 后统一推送
        （避免频繁打扰用户）
                │
                ▼
        推送通知给用户（桌面通知 / 移动端 Push）
        展示决策队列：逐条显示，支持批量处理
                │
                ▼
        用户逐条确认 / 拒绝（附原因）/ 修改后确认
                │
                ▼
        Leader 将用户决策写入自己的记忆
        并将结果下发给对应 Agent，Agent 继续执行
```

---

**单 Agent 模式决策流程**：

```
Agent 遇到疑问 或 产出成果物
        │
        ▼
Agent 查询自身记忆（他过去如何处理过类似情况）
        │
        ├── 记忆命中：自动处理，记录沿用日志
        │
        └── 记忆未命中
                │
                ▼
        Agent 直接向用户发起确认请求
        （通知 + Chat 中展示决策卡片）
                │
                ▼
        用户确认 / 拒绝 / 修改
                │
                ▼
        Agent 将用户决策写入自己的记忆，继续执行
```

---

**决策通知 UI 设计要点**：
- 批量决策时，以卡片列表形式展示，每张卡片包含：来源 Agent、决策类型、内容摘要、关联成果物（可展开）
- 支持"全部批准"一键操作
- Leader 自动处理的决策，单独以折叠卡片展示，用户可随时查看和覆盖
- 决策历史可在 Leader Agent 的记忆管理界面中查看和管理

---

**执行控制：任务终止**：
- 任务执行中，用户可随时点击"终止任务"按钮
- 终止操作会向所有正在运行的子任务发送停止信号
- 已完成的子任务结果保留，终止时刻的执行状态和产出写入 Trace 日志
- 终止后，用户可选择：查看已有产出 / 以当前结果为基础重新下发任务
- 任务完成（无论成功/失败/终止）后，通过推送通知告知用户，通知内容包含任务标题和完成状态摘要

#### 5.2.4 用户审批节点（Pipeline 阶段确认）

对于复杂任务或团队任务，执行过程往往是**串行的多阶段 Pipeline**（如：需求分析 → 产品设计 → 研发 → 测试）。每个阶段都可能产出成果物，用户可以决定是否在阶段间插入人工确认点。

**Pipeline 审批模式**（任务下发时配置，也可在任务执行中调整）：

| 模式 | 说明 | 适用场景 |
|-----|------|---------|
| `per_stage` 逐阶段确认 | 每个阶段完成后，成果物展示给用户审核，确认后才流转到下一阶段 | 对质量要求高、不熟悉 Agent 能力时 |
| `auto_approve` 全链路自动 | 所有阶段成果物自动放行，任务一气呵成，完成后统一通知 | 对 Agent 已充分信任，追求执行效率时 |
| `custom` 自定义 | 针对每个阶段单独设置是否需要确认，精细控制 | 部分阶段高风险、部分阶段低风险的混合场景 |

**逐阶段确认（per_stage）的交互流程**：

```
产品 Agent 完成需求文档
        │
        ▼
系统暂停，向用户推送通知
"[产品 Agent] 已完成「需求分析文档」，请审核后决定是否继续"
        │
        ▼
用户在 Chat / 看板中查看成果物
        │
        ├── ✅ 确认通过 → 成果物作为输入传给下一阶段 Agent（研发 Agent）
        ├── ✏️ 修改后确认 → 用户可直接编辑成果物内容，确认后流转
        └── ❌ 拒绝并说明原因 → 当前 Agent 收到反馈，重新执行本阶段
```

**高风险工具强制审批**（不受 Pipeline 模式影响）：
- 无论 Pipeline 设置为何种模式，涉及高风险工具（系统命令、外部 API 写操作）的操作**始终触发用户审批**，不可通过 `auto_approve` 跳过

### 5.3 圆桌会议与头脑风暴

#### 5.3.1 功能概述

**用户故事**：作为用户，我希望召集多个具有不同专业背景的 Agent 针对一个问题展开讨论，从多个视角获取洞察，并最终形成结论——就像召开一次专家圆桌会议或组织一场头脑风暴。

**与任务执行的本质区别**：

| 维度 | 任务执行（Task） | 圆桌讨论（Discussion） |
|-----|----------------|----------------------|
| 驱动方式 | 目标驱动，有交付物 | 过程驱动，探索性 |
| 结构 | DAG，有依赖关系 | 对话流，轮流发言 |
| Agent 角色 | 执行者 | 参与者 / 讨论者 |
| 工具使用 | 核心能力 | 默认关闭（可选开启） |
| 结束条件 | 目标完成 | 用户结束 / 达成共识 / 轮次上限 |
| 产出物 | 可交付的成果 | 讨论记录 + 结论摘要 |

#### 5.3.2 创建讨论会话

**创建入口**：主界面 Chat 视图或独立"讨论"页面，点击"发起圆桌"。

**配置步骤**：

**Step 1：选择参与者（必填，2–8 个 Agent）**
- 从 Agent 列表多选参与讨论的 Agent
- 参与者无角色约束：可以是同类型 Agent（如 3 个不同风格的设计师），也可以是不同职能 Agent（如产品 + 研发 + 测试）
- 显示每个 Agent 的角色简介，帮助用户理解他们的讨论视角

**Step 2：选择讨论模式（必填）**

| 模式 | 描述 | 适用场景 |
|-----|------|---------|
| 🔵 圆桌会议（Roundtable） | 各方平等发言，从各自专业视角分析议题，寻求共识 | 需求评审、方案决策、跨职能讨论 |
| 🟡 头脑风暴（Brainstorm） | 鼓励发散思维，不批判他人观点，积累尽可能多的想法 | 创意激发、方向探索、问题定义 |
| 🔴 辩论（Debate） | 参与者持对立立场互相论辩，充分暴露分歧和风险 | 投资决策、技术选型、风险评估 |

**Step 3：主持人设置（可选）**
- **无主持人**（默认）：Agent 按顺序轮流发言，用户可随时介入
- **指定主持人**：从参与者中选择一个 Agent 担任主持人，他负责调度发言顺序、总结阶段性观点、最终宣布结论
- **自动生成主持人**：系统创建一个轻量级主持人 Agent（仅含协调 prompt，无额外配置），专职协调会议

**Step 4：输入议题（必填）**
- 支持自然语言描述：问题、需求、场景、背景信息
- 支持附件（文件、图片）作为讨论背景材料
- 支持 `@AgentName` 给特定 Agent 预置初始立场（如 `@价值投资专家 你认为这只股票被低估`）

**Step 5：高级配置（折叠，选填）**
- 每 Agent 最大发言轮数（默认 10 轮）
- 是否在讨论中允许 Agent 使用工具（默认关闭；开启后 Agent 可调用白名单内工具辅助论点）
- 是否在讨论结束后自动写入各 Agent 记忆（默认开启）

#### 5.3.3 讨论进行中

**发言流程（无主持人 · 顺序模式）**：

```
用户提交议题
       │
       ▼
Agent 1 发言（从其专业视角出发，引用自身记忆中相关经验）
       │
       ▼
Agent 2 发言（可以回应 Agent 1 的观点，也可独立阐述）
       │
       ▼
Agent 3 发言（同上）
       │
       ▼
...（轮流，直到用户结束 或 达到最大轮次）
       │
       ▼
[触发结论生成] → 合成总结，写入各 Agent 记忆
```

**发言流程（有主持人 · 协调模式）**：

```
用户提交议题
       │
       ▼
主持人 Agent 宣布议程，指定第一个发言者
       │
       ▼
被指定 Agent 发言
       │
       ▼
主持人 Agent 回应（总结观点，提出追问，指定下一发言者）
       │
       ▼
...（主持人持续调度，保持讨论焦点）
       │
       ▼
主持人 Agent 宣布各方已充分表达 → 触发结论生成
```

**实时 UI 交互**：
- 讨论流以多人聊天界面呈现：每条发言带有 Agent 头像 + 名称 + 角色标签
- 每条发言支持流式输出，实时显示 Agent 的思考过程
- 当前发言的 Agent 在参与者列表中高亮显示（"正在发言..."）
- 已等待的 Agent 显示"排队中..."状态
- 发言内容支持完整 Markdown 渲染

**用户介入方式**：

用户在讨论进行中可以随时：
- **插话**：在输入框发送消息，内容作为"用户补充"插入讨论流，所有 Agent 后续发言都会考虑该内容
- **定向提问**：`@AgentName 你对刚才的观点怎么看？` 直接向特定 Agent 发起追问
- **重定向**：`大家聚焦一下 X 方向` 引导讨论方向
- **暂停**：暂停讨论，保存当前状态，稍后继续
- **跳过**：跳过当前 Agent 的发言轮次
- **提前结束**：立即触发结论生成，不等待剩余轮次

#### 5.3.4 Agent 发言质量保障

**防止"回音壁"效应**：每个 Agent 的发言 prompt 中明确要求：
- 从自身专业立场出发，可以赞同或质疑他人观点，但必须提供独立的论据
- 不能简单重复他人观点，须有新的视角或补充
- 引用讨论历史时须明确标注来源 Agent

**记忆注入**：讨论开始时，每个 Agent 自动检索自身记忆中与议题相关的内容，作为隐式背景知识融入其发言，使有经验的 Agent 能在讨论中体现出"资深感"。

**上下文裁剪**：当讨论历史过长时，系统自动将早期内容生成滚动摘要，最近 N 条发言以完整形式提供给 Agent，保证发言相关性不下降。

#### 5.3.5 讨论结束与结论生成

**触发条件**（满足任意一个）：
- 用户手动触发"结束讨论"
- 每位 Agent 达到最大发言轮数
- 主持人 Agent 判断已达成共识并宣布结束

**结论生成流程**：
1. 系统调用独立的 LLM 请求，基于完整讨论记录生成结构化结论
2. 结论包含：各方核心观点摘要、争议点与分歧、形成的共识、推荐的行动方向
3. 结论展示在讨论界面底部，支持用户编辑和补充
4. 结论可导出为 Markdown 文档

**记忆写入**：
- 讨论结束后，系统为每位参与 Agent 从讨论中提取关键洞察，写入各自的记忆
- 写入内容包括：本次讨论的核心结论、他人的有价值观点、形成的共识
- 写入过程异步进行，不阻塞用户查看结论

**验收标准**：
- 3 个 Agent 可在 5 分钟内完成一轮完整的圆桌讨论并生成结论
- 每个 Agent 的发言明显体现其角色特征，不出现"所有 Agent 说同样内容"的情况
- 用户插话后，后续所有 Agent 的发言都能体现对该插话内容的回应
- 讨论结束后能在 Agent 记忆列表中查到本次讨论来源的记忆条目

#### 5.3.6 讨论历史管理

- 所有讨论会话持久化保存，可在"讨论历史"页面查看
- 支持搜索：按参与 Agent、关键词、时间范围筛选
- 历史讨论可以"续会"：基于已有讨论记录继续深入，而不是从头开始
- 可将讨论记录 + 结论导出为 Markdown / PDF

### 5.6 Hub 管理（面向管理员）

#### 5.6.1 Hub 控制台

- 系统状态总览：CPU / 内存使用率、活跃任务数、在线 Client 数
- 租户管理：创建 / 停用租户、配置资源配额
- 用户管理：邀请用户、禁用账号、重置密码、分配角色
- 模型中心：管理员统一维护 Provider、模型目录、模型档案和租户默认值
- MCP 服务注册与管理：统一注册可用的 MCP 服务供租户选择

#### 5.6.2 模型中心

模型管理采用四层对象：

1. `ModelProvider`：管理员维护 Provider 连接、端点、密钥引用和状态。
2. `ModelCatalogItem`：维护某个 Provider 下可用的原始模型目录，区分 `inference` 和 `embedding`。
3. `ModelProfile`：将原始模型包装为业务可消费的稳定档案，固化名称、说明和参数预设。
4. `TenantModelPolicy`：控制租户允许使用的模型档案，以及默认的 Agent / Summary / Embedding 绑定。

产品行为要求：

1. 普通成员不直接填写 API Key。
2. Agent 创建页只选择 `ModelProfile`。
3. 讨论结论、记忆提取等系统级摘要能力使用租户默认 `Summary` 档案。
4. Embedding 使用租户默认 `Embedding` 档案，不由每个 Agent 单独配置。

#### 5.6.3 多租户隔离

每个租户（Tenant）拥有独立的：
- Agent 库 + 团队配置（含每个 Agent 的完整三维度：Identity / Capability / Memory）
- 模型档案可见性与默认绑定策略
- 向量记忆存储（每个 Agent 的记忆严格隔离）
- 任务历史 + 执行日志
- MCP 绑定配置

租户间数据严格隔离，不可跨租户访问，租户级别可配置资源配额（最大 Agent 数、最大并发任务数、存储容量）。

对应现实中"公司 / 部门 / 组织"概念：

- 一个租户下可创建多个团队组
- 团队组包含多个团队 + 多个用户

## 7. 权限与安全模型

### 7.1 Agent 能力白名单

**设计原则**：默认无权限，所有能力需显式授权（Allowlist 模型）。

| 风险等级 | 工具类型 | 默认状态 | 额外要求 |
|---------|---------|---------|---------|
| 🟢 低风险 | 文件读取、数据处理、Markdown 渲染 | 可默认勾选 | 无 |
| 🟡 中风险 | 文件写入、HTTP 请求、代码执行（沙箱） | 需显式勾选 | 无 |
| 🔴 高风险 | 系统命令执行、进程管理、外部 API 写操作 | 需显式勾选 | 触发审批节点（不可关闭） |

**讨论中工具使用的额外说明**：
- 讨论模式下，Agent 默认不使用任何工具（避免讨论跑偏为任务执行）
- 用户可在创建讨论时开启"工具增强"，允许 Agent 在发言中调用低风险工具（如 web_search）辅助论证
- 高风险工具在讨论模式下始终禁用

### 7.2 Hub RBAC 权限模型

**系统预置角色**：

| 角色 | 权限范围 |
|-----|---------|
| `hub_admin` | Hub 全部管理权限，含租户创建与管理 |
| `tenant_admin` | 本租户内全部管理权限 |
| `team_manager` | 管理指定团队的 Agent 配置、任务下发、成员管理 |
| `member` | 创建 Agent、下发任务、查看自己的执行结果 |
| `viewer` | 只读：查看任务结果、Agent 状态（不可操作） |
| `discussion.create` | 创建讨论会话 |
| `discussion.view` | 查看讨论记录 |
| `discussion.conclude` | 结束讨论 |

租户管理员可在预置角色基础上创建自定义角色，细粒度配置各资源的读写权限。

### 7.3 认证安全

- Token 存储使用系统级安全存储（macOS Keychain / Windows Credential Store / Android Keystore）
- 所有 Client ↔ Hub 通信使用 HTTPS / TLS 加密
- 支持 Token 手动吊销（Hub 管理员可强制登出指定 Client）
- 登录失败次数限制 + 账号临时锁定机制
- 支持 2FA 双因子认证

### 7.4 数据安全

- Hub 本地数据库支持静态加密（AES-256）
- Agent 记忆库向量数据加密存储（记忆属于 Agent 个人，加密保护其隐私性）
- 执行日志自动脱敏（不记录 LLM API Key 等敏感信息）
- 私有化部署时，所有数据（对话、Agent 记忆、任务日志）完全留存在客户自有环境，Octopus 无法访问

### 8.1 性能指标

| 指标 | 目标值 |
|-----|--------|
| Agent 首次响应时间 | < 2s（不含 LLM 推理时间） |
| 任务看板加载时间 | < 1s |
| 单 Hub 并发任务支持 | ≥ 50 个并发任务 |
| Agent 记忆检索延迟（RAG） | < 500ms |
| 桌面客户端冷启动时间 | < 3s |
| 流式输出首 Token 延迟 | < 500ms（不含 LLM 网络延迟） |
| 讨论会话首次发言延迟 | < 3s（含记忆检索） |
| 结论生成时间 | < 10s（20 轮以内的讨论） |

### 8.2 可靠性

- Hub 服务目标可用性：99.5%（私有化部署，不含客户基础设施故障）
- 任务执行失败后自动保存状态，支持断点续跑
- 子任务失败不中断其他并行子任务的执行
- 网络断开时 Client 展示明确离线提示，本地缓存历史任务结果可查看

### 8.3 可扩展性

- LLM Provider 插件化接口，新增 Provider 无需修改核心代码
- MCP 标准协议接入，支持接入任意 MCP Server
- Agent 工具体系开放扩展接口（自定义 Tool 注册）
- 支持 Hub 集群化横向扩展

## 附录：术语表

| 术语 | 定义 |
|-----|------|
| **Octopus** | 本产品全称，多 Agent 协作操作系统 |
| **Hub** | 系统执行层，负责 Agent 运行、任务调度、讨论引擎、数据存储、权限管理 |
| **Client** | 用户交互层，负责 UI 渲染，无状态，可连接多个 Hub |
| **Agent** | 对现实世界"人"的数字化抽象，拥有 Identity（身份）、Capability（能力）、Memory（记忆）三个维度 |
| **Agent Identity** | Agent 的身份维度：名字、头像、角色描述、性格标签、System Prompt |
| **Agent Capability** | Agent 的能力维度：工具白名单、Skill 附加、MCP 绑定 |
| **Agent Memory** | Agent 的记忆维度：通过任务和讨论积累的个人经验，属于 Agent 私有 |
| **Leader Agent** | 团队中负责任务分解和协调分配的特殊 Agent |
| **Moderator Agent** | 讨论会话中负责调度发言顺序、总结观点、宣布结论的主持人 Agent |
| **Team** | 多个 Agent 组成的协作单元，用于执行任务 |
| **DiscussionSession** | 圆桌会议 / 头脑风暴会话，多个 Agent 围绕议题进行迭代讨论的实体 |
| **DiscussionTurn** | 讨论会话中的一次发言记录（Agent 发言或用户插话） |
| **Roundtable** | 圆桌讨论模式：各方平等发言，从专业视角分析 |
| **Brainstorm** | 头脑风暴模式：鼓励发散，不批判，积累想法 |
| **Debate** | 辩论模式：参与者持对立立场互相论辩 |
| **TurnStrategy** | 讨论中的发言调度策略：Sequential（顺序）或 Moderated（主持人指定） |
| **TeamGroup** | 多个 Team 组成的组织单元，对应现实中的"部门 / 公司" |
| **Task** | 用户下发的一次工作指令，包含完整执行生命周期 |
| **Subtask** | Task 被 Leader 分解后分配给具体 Agent 的子任务 |
| **Routing Rules** | 用户定义的任务分配规则，优先级高于 LLM 决策 |
| **Approval Gate** | 任务执行流程中需要用户手动确认的暂停节点 |
| **MCP** | Model Context Protocol，标准化 AI 工具接入协议 |
| **ModelProvider** | Hub 管理员维护的模型 Provider 连接对象，包含端点、密钥引用和状态 |
| **ModelCatalogItem** | 某个 Provider 下的原始模型目录项，区分推理和 Embedding |
| **ModelProfile** | 面向业务可选择的模型档案，引用目录项并固化参数预设 |
| **TenantModelPolicy** | 租户允许使用的模型档案集合，以及默认 Agent / Summary / Embedding 绑定 |
| **RAG** | Retrieval-Augmented Generation，检索增强生成 |
| **Skill** | 可附加到 Agent 的能力模块，包含 Prompt 片段 + 工具授权 + MCP 绑定 |
| **Whitelist** | Agent 能力白名单，明确列出该 Agent 可使用的工具集合 |
| **Tenant** | Hub 上的租户，拥有独立数据空间和用户组 |
| **RBAC** | Role-Based Access Control，基于角色的访问控制 |
| **Decision** | Agent 执行过程中提交给 Leader 或用户的决策请求 |
| **Pipeline** | 多阶段串行任务链，每个阶段产出成果物后可插入确认节点 |
| **Trace** | 任务执行的完整日志，包含每步 Agent 的输入输出和工具调用 |

## Design Note

- Problem:
  - The tracked tree already proves remote connection, session handling, and project-scoped workbench surfaces, but authenticated remote desktop users still have no truthful path to discover existing projects in the current workspace or remember which project to reopen.
- Goal:
  - Add one workspace-scoped project discovery and entry surface so remote desktop users can select an existing project after login, persist that non-secret choice locally, and return directly to the current project workbench when the remembered project remains valid.
- Acceptance Criteria:
  - Desktop remote bootstrap resolves to a project list when authenticated but no project is remembered.
  - Project selection leads into the existing project-scoped `Tasks` route family.
  - Remembered project ids are reused only when the project still exists in the current workspace.
  - Local mode remains unchanged.
- Non-functional Constraints:
  - No changes to auth schemas or login flow semantics.
  - No project CRUD or workspace discovery in this slice.
  - No new cross-language DTO beyond the existing `Project`.
- MVP Boundary:
  - One workspace-scoped project list query.
  - One `ProjectsView` surface.
  - One optional remembered `projectId` persisted with the desktop connection profile.
- Layer Placement:
  - `crates/domain-context` owns workspace-scoped project persistence and ordered reads.
  - `crates/runtime` owns the public project-list query surface and preserves context truth.
  - `apps/remote-hub` exposes the authenticated workspace-scoped HTTP read route.
  - `apps/desktop/src-tauri` maps the shared local command to the runtime query without introducing app-local DTO truth.
  - `packages/schema-ts` and `packages/hub-client` own the shared parser and transport parity for `Project[]`.
  - `apps/desktop/src` owns project entry routing, remembered project persistence, and the `Projects` view composition.
- Module Boundaries:
  - `ConnectionsView` remains the authority surface for connection/session state and must not absorb project selection.
  - `ProjectsView` is the only new remote project-selection surface in this slice.
  - Desktop bootstrap may consult the remembered `projectId`, but runtime remains the source of existence truth.
  - App-local persistence may remember `projectId`, but remote access tokens remain memory-only.
- Inputs:
  - Existing `Project` schema, project-scoped workbench routes, desktop connection store, hub store, and local/remote transport parity patterns.
  - Existing remote-hub auth enforcement via workspace membership checks.
- Outputs:
  - One shared read method: `HubClient.listProjects(workspaceId)`.
  - One shared local transport command: `hub:list_projects`.
  - One remote route: `GET /api/workspaces/:workspaceId/projects`.
  - One desktop `Projects` route and view.
  - One bootstrap rule set for remembered or missing project scope.
- State Transitions:
  - `remote authenticated + no remembered project -> ProjectsView`
  - `remote authenticated + remembered valid project -> existing project Tasks route`
  - `remote authenticated + remembered stale project -> clear remembered project and return to ProjectsView`
  - `local -> existing demo workbench route`
- Error Handling:
  - Unauthorized or workspace-forbidden remote responses keep the shell on `/connections` through existing auth handling.
  - Project list fetch failures surface through the existing shell error banner.
  - Stale remembered project ids are cleared only after runtime truth proves they no longer exist in the current workspace.
- Tech Stack Decision:
  - No new stack. Reuse the existing Rust + schema-ts + hub-client + Vue/Pinia/Tauri stack.
- Visual Framework Impact:
  - Yes. Slice 17 adds a dedicated `Projects` surface and adjusts left-rail navigation so project-scoped links appear only when a current project exists.
- Human Approval Points:
  - None.
- Reused Components:
  - Existing `Project` schema
  - Existing `HubClient` abstraction
  - Existing remote auth/session handling and desktop shell routing patterns
  - Existing local-host and remote-hub parity implementation style from `listRuns` and `getProjectKnowledge`
- New Abstractions:
  - One workspace-scoped project list query across runtime and transport layers.
  - One app-local `ProjectsView`.
- Trade-offs:
  - Remembering only `projectId` keeps persistence small and non-secret, but requires a runtime existence check before direct entry.
  - Reusing the existing `Tasks` route family avoids inventing a remote-only workspace dashboard, but adds one more bootstrap branch that must stay explicit in tests.
- Test Strategy:
  - Red-first Rust tests for workspace-scoped project list ordering, empty results, and workspace isolation.
  - Red-first transport parity tests for `HubClient.listProjects`.
  - Red-first desktop tests for authenticated remote entry without a remembered project, remembered project restore, stale remembered project recovery, and local-mode regression.
- ADR Needed:
  - No, unless implementation forces a new durable rule for desktop persistence scope or project ownership boundaries.

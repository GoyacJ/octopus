# octopus 视觉框架与信息架构基线

更新时间：2026-03-24  
文档状态：Draft  
文档定位：首版视觉框架与页面分层说明  
关联文档：`docs/PRD.md`、`docs/SAD.md`、`docs/DEVELOPMENT_STANDARDS.md`

---

## 1. 文档目的

本文档用于在 UI 正式编码前，锁定 `octopus` 首版的表面职责、信息架构、页面优先级和视觉语法，避免 AI 在缺少界面边界时生成风格漂移或导航混乱的前端骨架。

---

## 2. 设计方向

`octopus` 的首版视觉方向遵守以下原则：

1. 现代简约。
2. 冷静克制。
3. 强信息层级。
4. 连续一致的跨端语法。
5. 面向生产治理，不做娱乐化或聊天玩具化表达。

实现要求：

1. 以中性色为主基底，品牌强调色保持节制。
2. 高风险、等待、审批、失败、恢复等状态必须有清晰语义色与层级。
3. 不依赖厚重阴影、强渐变或花哨动效制造“高级感”。
4. `system`、`light`、`dark` 与 `zh-CN`、`en-US` 必须共享同一套结构与视觉语法。

---

## 3. 表面职责

| 表面 | 角色 | 首版职责 |
| --- | --- | --- |
| Web | 主控制面 | 工作区、agent、run、审批、审计、扩展治理、导入导出 |
| H5 / PWA | 轻控制面 | 通知跳转、审批、状态查看、轻量交互 |
| Desktop | 强控制面 + 本机节点 | 复用 Web UI，增加本机能力与系统集成 |
| Mobile | 控制优先 | 登录、审批、ask-user、通知、状态查看、轻量接管 |

---

## 4. 首版信息架构

Web / Desktop 的一级导航按能力域组织，不按技术实现细节组织：

1. `Overview`
2. `Workspaces`
3. `Agents`
4. `Runs`
5. `Inbox`
6. `Triggers`
7. `Artifacts`
8. `Extensions`
9. `Nodes`
10. `Audit`
11. `Settings`

说明：

1. `Inbox` 统一承载 `ask-user`、审批和待恢复事项，不拆成多个分散入口。
2. `Extensions` 下再分 `Built-in Tools`、`Skills`、`Prompt Packs`、`MCP Servers`、`Plugins`。
3. `Runs` 是首版核心导航之一，必须优先于“聊天式入口”建设。

---

## 5. 首批页面优先级

首轮 UI 落地只允许进入以下页面：

1. 应用壳与全局导航。
2. 工作区切换与基础概览。
3. agent 列表与详情壳。
4. run 列表、run 详情、时间线视图。
5. inbox 列表与处理页。
6. 审计视图基础壳。

暂不进入：

1. 复杂可视化仪表盘。
2. 自定义工作流搭建器。
3. 高频营销页面或聊天入口优化。
4. 重资产 plugin marketplace 体验。

---

## 6. 布局语法

### 6.1 桌面 / Web

1. 左侧为一级导航。
2. 顶部为当前工作区、全局检索、主题、语言和身份操作。
3. 主内容区优先采用列表 + 详情、时间线 + 侧栏、表单 + 审计上下文等治理型布局。

### 6.2 H5 / Mobile

1. 以任务和待办优先，不复制桌面信息密度。
2. 审批、通知跳转、状态查看和轻量接管优先级高于管理配置。
3. 长时间线与复杂配置默认折叠或分步呈现。

---

## 7. 组件与文案边界

1. 业务页面只能消费 `packages/ui` 暴露的自建组件。
2. UI copy 必须进入 `packages/i18n`，禁止在业务组件内硬编码中英文。
3. 颜色、间距、圆角、阴影、动效只允许来自 `packages/design-tokens`。
4. 状态标签、风险提示、审批动作、失败解释必须使用统一语义组件。

---

## 8. 文档维护规则

以下变更必须同步更新本文档：

1. 一级导航调整。
2. 核心页面优先级调整。
3. 表面职责变化。
4. 主题、视觉语法或文案策略变化。

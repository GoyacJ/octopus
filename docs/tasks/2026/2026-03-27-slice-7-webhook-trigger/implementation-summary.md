## Implementation Summary

- Goal:
  - Add the minimum governed webhook ingress so authenticated external events can create deduped trigger deliveries and governed automation runs.
- Files Added:
  - `crates/runtime/tests/slice7_webhook_trigger.rs`
  - `apps/remote-hub/tests/webhook_ingress.rs`
- Files Changed:
  - `crates/runtime/migrations/0006_slice6_trigger_expansion.sql` and `schemas/runtime/trigger.schema.json` to persist webhook ingress metadata and secret-presence contract fields
  - `crates/runtime/src/models.rs`, `crates/runtime/src/lib.rs`, and `crates/runtime/src/services.rs` to add typed webhook config, hashed-secret persistence, one-time secret reveal, and `dispatch_webhook_event(...)`
  - `apps/remote-hub/src/lib.rs` to add `POST /api/triggers/{trigger_id}/webhook`, required-header extraction, and runtime error mapping
  - `Cargo.toml` and `crates/runtime/Cargo.toml` to register `sha2`
- Files Removed:
  - None.
- Structure Decision:
  - Keep HTTP ingress thin in `apps/remote-hub` and let the runtime own trigger-type checks, secret validation, idempotency, and governed delivery projection.
- Why This Structure:
  - Webhook ingress is an external boundary, so request validation must stay narrow and auditable while all business and governance logic remains in the runtime.
- Reused Patterns:
  - The shared trigger-substrate dispatch path plus existing delivery dedupe, reopen, approval, and deny semantics.
- New Dependencies:
  - `sha2`
- Error Handling Strategy:
  - Missing `Idempotency-Key` is rejected as a bad request, invalid secrets are rejected as unauthorized, duplicate ingress reuses the same delivery / run, and approval / deny outcomes still flow through the existing delivery state machine.
- Deferred Items:
  - No secret-rotation surface, no partner-specific webhook adapters, and no richer remote-hub auth / deployment model.
- Non-goals Preserved:
  - No desktop or webhook-management UI, no partner-specific adapters, and no real connector ecosystem work.

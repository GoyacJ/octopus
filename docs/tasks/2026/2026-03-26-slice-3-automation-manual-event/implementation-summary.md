## Implementation Summary
- Goal:
  - Implement the first verified local automation slice for `manual_event` TriggerDelivery on top of the existing governed runtime.
- Files Added:
  - `crates/runtime/migrations/0003_slice3_automation.sql`

  - `docs/decisions/0003-automation-delivery-projects-into-derived-tasks.md`

- Files Changed:

  - `crates/observe-artifact/src/lib.rs`

  - `schemas/runtime/*`
  - `schemas/observe/trace-record.schema.json`
  - repository entry docs and index docs
- Files Removed:
  - None.
- Structure Decision:
  - Keep Automation, Trigger, and TriggerDelivery persistence/orchestration inside `crates/runtime`, reuse the existing TaskIntake and RunOrchestrator chain, and keep observation/governance crates unchanged except for new trace constants.
- Why This Structure:
  - It preserves one governed execution path and avoids inventing a second runtime model for automation before scheduler or transport slices exist.
- Reused Patterns:
  - Existing Slice 1/2 runtime, migration, and contract-test patterns.
- New Dependencies:
  - None planned.
- Error Handling Strategy:
  - Delivery dedupe reuses the existing persisted delivery and run when the same `dedupe_key` is seen again.
  - Delivery retry is explicit and only allowed when the underlying Run remains retryable.
  - Approval and deny results sync back into TriggerDelivery state without bypassing the existing approval lifecycle.
- Deferred Items:
  - `cron`, `webhook`, `MCP event`, scheduler work, and Shared Knowledge remain deferred.
- Non-goals Preserved:
  - No app surfaces, transport, or knowledge-system implementation is introduced.

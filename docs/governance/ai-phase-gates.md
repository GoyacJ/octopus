# AI Phase Gates

## 1. 目的

本文件定义 AI 在本仓库中进行设计、编码、重构、测试、补文档时必须遵守的阶段控制机制。

目标：

- 防止 AI 在需求不清时直接编码
- 防止范围漂移、过度抽象、越层实现
- 防止“看起来完成”但缺少契约、测试、说明的交付
- 为 AI 自检、人工评审、PR 审核提供统一阶段门

本文件与以下文档配合使用：

- `AGENTS.md`
- `ai-engineering-guidelines.md`
- `ai-delivery-templates.md`
- `code-review-checklist.md`

---

## 2. 使用原则

### 2.1 主线阶段

默认按以下主线推进：

`Phase 0 -> Phase 1 -> Phase 2 -> Phase 3 -> Phase 4 -> Phase 5`

### 2.2 允许回退

阶段控制不是僵硬瀑布流。  
当后续阶段发现前置阶段结论不成立时，必须回退：

- 测试暴露结构问题，可回退到 Phase 3 或 Phase 1
- 契约不完整，可回退到 Phase 2
- 范围理解有误，可回退到 Phase 0

### 2.3 阶段门组成

每个阶段都必须定义：

- **入口条件**
- **阶段内 checklist**
- **必须产物**
- **Hard Stop**
- **出口标准**

只有通过出口标准，才允许进入下一阶段。

### 2.4 边界、验收与审批的责任归属

- `Goal`、`Scope`、`Out of Scope`、`Acceptance Criteria`、`Human Approval Points` 属于人或仓库事实源先定义的边界
- AI 可以帮助澄清、拆解和补充说明，但不得私自扩大范围、降低验收门槛、跳过审批点，或把自己的判断当作批准结果
- 当边界、验收或审批归属不清时，必须停在阶段门，而不是继续实现

### 2.5 文档分层与适用范围

- 产品语义、正式范围、release slicing 变化进入 `docs/product/PRD.md`
- 任务级目标、验收标准、非功能约束、MVP 边界与设计说明进入 task package 与相应模板
- 共享契约、事件、DTO、状态语义继续遵守 schema-first
- `docs/architecture/VISUAL_FRAMEWORK.md` 只约束 UI / Surface 任务，不得成为纯后端、契约或治理任务的全局前置门

## 3. Phase 0：问题定义与边界冻结

## 3.1 目标

确保本次任务的目标、范围、非范围、验收标准、非功能约束、验证方法与人工审批点已清楚，避免 AI 在问题不明确时直接编码。

## 3.2 入口条件

满足以下条件才允许进入 Phase 0：

- 已收到明确任务或需求
- 已知本次任务是新增、修改、重构、补测试或补文档中的哪一种
- 已知当前任务关联的模块或领域

## 3.3 Checklist

- 本次任务的目标是什么
- 本次任务属于哪个模块或子域
- 本次任务是新增、修改、重构、修复还是补测试
- 本次任务影响哪些层
- 是否涉及 schema 变更
- 是否涉及公共接口变更
- 是否涉及状态机变更
- 是否涉及权限 / 审批 / 预算 / 安全边界
- 本次任务的验收标准是什么
- 本次任务的非功能约束是什么
- 本次任务的最小可交付 MVP slice 是什么
- 是否存在必须由人确认的 approval points
- 本次结论应落到哪一层事实源：PRD、task package、schema / contract、ADR 还是其他 owner doc
- 若是 UI / Surface 任务，是否需要引用或补充 `docs/architecture/VISUAL_FRAMEWORK.md`
- 本次明确不做什么
- 如何验证本次任务完成

## 3.4 必须产物

必须至少产出一份任务定义说明，可使用 `Task Definition` 模板。

## 3.5 Hard Stop

满足以下任一情况，禁止进入后续阶段：

- 目标不清楚
- 范围和非范围未区分
- 影响模块不明确
- 验收标准不明确
- 涉及关键非功能约束但未标注
- 涉及人工审批但审批点未明确
- 验证方式不明确
- 文档归属不明确
- 明显存在范围漂移但未标注

## 3.6 出口标准

至少明确以下内容：

- Goal
- Scope
- Out of Scope
- Acceptance Criteria
- Non-functional Constraints
- MVP Boundary
- Human Approval Points
- Affected Modules
- Risks
- Validation

## 4. Phase 1：设计与方案阶段

## 4.1 目标

在编码前形成最小必要设计，明确边界、分层、依赖、输入输出、失败路径、MVP 边界与测试策略。

## 4.2 入口条件

- Phase 0 已通过
- 范围与目标已冻结
- 关键风险已初步识别

## 4.3 Checklist

- 代码应该属于哪一层
- 模块边界是什么
- 是否需要新增模块
- 输入输出是什么
- 状态流转是什么
- 错误路径是什么
- MVP slice 边界是什么
- 非功能约束如何落地
- 若新增或变更技术栈，决策是什么，为什么
- 是否涉及 UI / Surface；若涉及，`VISUAL_FRAMEWORK.md` 对当前任务是否已足够
- 哪些点必须等待人工确认才能继续
- 哪些依赖需要复用
- 是否真的需要新抽象
- 是否可能形成大文件或大类
- 测试策略是什么
- 是否需要 ADR

## 4.4 必须产物

至少产出以下之一：

- 简版设计说明
- 模块边界说明
- 接口草图
- schema 草图
- 状态机草图

建议使用 `Design Note` 模板。

## 4.5 Hard Stop

满足以下任一情况，禁止进入实现：

- 没有明确层归属
- 没有明确输入输出
- 没有错误路径说明
- 没有明确 MVP 边界
- 涉及技术栈新增或变更但没有决策说明
- UI / Surface 任务需要视觉冻结但未说明依据
- 人工审批点或人工确认点不明确
- 没有测试思路
- 明显引入过度抽象但无理由

## 4.6 出口标准

- 层归属明确
- 模块边界明确
- 输入输出明确
- 错误路径明确
- MVP 边界明确
- 非功能约束落地方式明确
- 技术栈决策在需要时已明确
- UI / Surface 任务的视觉依据在需要时已明确
- 测试策略明确
- 人工审批点已明确
- 是否需要 schema / ADR 已明确

---

## 5. Phase 2：契约与结构阶段

## 5.1 目标

先冻结共享契约、目录归属与模块骨架，再进入具体实现。

## 5.2 入口条件

- Phase 1 已通过
- 已知要新增或修改哪些对象、接口或事件

## 5.3 Checklist

- 是否需要先修改 `schemas/`
- 是否涉及跨 Rust / TypeScript 共享对象
- 是否新增事件 / DTO / command / query / domain model
- 目录归属是否正确
- 是否会破坏既有分层
- 是否需要创建模块骨架
- 是否需要 ADR
- 是否存在兼容性影响
- 是否需要迁移说明

## 5.4 必须产物

根据任务类型，至少产出以下之一：

- schema 变更
- 接口定义
- 模块骨架
- 目录归属说明
- 兼容性说明
- ADR 草案或 ADR 结论

建议使用 `Contract Change` 模板。

## 5.5 Hard Stop

满足以下任一情况，禁止进入实现：

- 涉及共享契约但未更新 schema
- 涉及公共接口但未说明兼容性
- 模块归属不明确
- 依赖方向不明确
- 需要 ADR 但未记录结论

## 5.6 出口标准

- schema / 接口 / 模块骨架已落位
- 命名已确定
- 目录归属正确
- 兼容性影响已标注
- 依赖方向正确

---

## 6. Phase 3：实现阶段

## 6.1 目标

在既定边界内完成实现，不越层、不漂移、不堆积坏结构。

## 6.2 入口条件

- Phase 2 已通过
- 共享契约与模块归属已明确
- 已知本次实现不应超出的范围
- 已知本次实现对应的 MVP slice 与人工审批点

## 6.3 Checklist

- 是否遵守分层
- 是否保持在已冻结的 MVP slice 内
- 是否复用已有模式
- 是否控制文件大小
- 是否控制函数长度
- 是否避免 God Object / God Service
- 是否做了显式错误处理
- 是否去除了魔法值
- 是否补了必要注释
- 是否避免无关改动
- 是否保留了清晰依赖方向
- 是否在触达人工审批点时停止并显式升级

## 6.4 必须产物

- 实现代码
- 必要测试代码
- 必要文档更新
- 实现说明

建议使用 `Implementation Summary` 模板。

## 6.5 Hard Stop

满足以下任一情况，禁止标记实现完成：

- 新增逻辑无测试计划或测试代码
- 超出已冻结 MVP 边界
- 文件明显过大但无拆分说明
- 存在跨层穿透
- 引入临时方案但未标注
- 触达人工审批点却继续推进
- 修改公共接口语义但未说明
- 明显存在无关重构

## 6.6 出口标准

- 核心实现已完成
- 编译或静态检查通过
- 结构符合分层
- 主要文件体积可控
- 关键风险已记录
- 未越出已冻结的 MVP 边界

---

## 7. Phase 4：测试与验证阶段

## 7.1 目标

验证本次实现不仅可运行，而且可回归、可维护、可解释。

## 7.2 入口条件

- Phase 3 已通过
- 核心实现已稳定

## 7.3 Checklist

- 是否有单元测试
- 是否有失败路径测试
- 是否有边界条件测试
- 是否有最小集成测试
- 是否覆盖了回归风险
- 是否验证了错误处理
- 是否验证了公共接口行为
- 是否验证了兼容性影响

## 7.4 必须产物

- 测试结果
- 验证说明
- 剩余缺口说明

建议使用 `Verification` 模板。

## 7.5 Hard Stop

满足以下任一情况，禁止进入交付：

- 只有 happy path 测试
- 没有失败路径测试
- 没有边界条件测试
- 测试明显依赖实现细节过多
- 关键路径未验证
- 回归风险点未验证

## 7.6 出口标准

- 核心路径已验证
- 失败路径已验证
- 边界条件已验证
- 测试结果可复现
- 剩余风险已记录

---

## 8. Phase 5：收口与交付阶段

## 8.1 目标

确保本次任务可以被安全合并、被后续接手、被清晰理解。

## 8.2 入口条件

- Phase 4 已通过
- 核心实现与测试已完成

## 8.3 Checklist

- 改了什么
- 为什么这样改
- 风险点是什么
- 是否有临时方案
- 后续要收敛什么
- 是否更新了文档
- 是否需要补 ADR
- 是否需要补迁移说明
- 验证结论是否附带可追溯证据
- 代码、测试、文档是否一致

## 8.4 必须产物

- 交付说明
- 风险说明
- 后续建议
- 关联文档或 ADR 更新

建议使用 `Delivery Note` 模板。

## 8.5 Hard Stop

满足以下任一情况，禁止标记完成：

- 无风险说明
- 临时方案未标注
- 验证结论缺少可追溯证据
- 文档、测试、代码不一致
- 明显影响公共行为但无说明
- 缺少后续待收敛项说明

## 8.6 出口标准

- 交付说明完整
- 风险透明
- 剩余问题透明
- 验证证据已留痕
- 文档更新完成
- 本次任务具备可评审、可合并状态

---

## 9. 阶段速查表

| Phase | 名称 | 核心目标 | 必须产物 |
|---|---|---|---|
| 0 | 问题定义与边界冻结 | 明确目标、范围、验收与前置边界 | Task Definition |
| 1 | 设计与方案 | 明确边界、分层、MVP、错误路径、测试策略 | Design Note |
| 2 | 契约与结构 | 先冻结 schema、接口、目录归属 | Contract Change / ADR / Skeleton |
| 3 | 实现 | 在既定边界内完成实现 | Implementation Summary |
| 4 | 测试与验证 | 证明实现可运行、可回归 | Verification |
| 5 | 收口与交付 | 形成可评审、可合并产物 | Delivery Note |

---

## 10. 一句话总纲

AI 在本仓库中开发时，不得跳过阶段门。  
任何实现都必须先明确问题，再明确设计，再冻结契约与结构，之后才允许实现、验证与交付。

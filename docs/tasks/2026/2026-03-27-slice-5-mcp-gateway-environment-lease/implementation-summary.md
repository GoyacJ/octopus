- Structure Decision:
  - Add a dedicated `crates/interop-mcp` workspace member and compose it into runtime rather than embedding connector persistence and fake gateway logic directly into `crates/runtime`.
- Why This Structure:
  - It keeps interop-specific records and execution behavior separate from runtime orchestration while preserving one governed run shell.
- Reused Patterns:
  - Reuse the existing SQLite-backed crate/store pattern, run-orchestrator execution shell, and contract-test structure.
- New Dependencies:
  - No new third-party crates were introduced; runtime now depends on the new in-repo `octopus-interop-mcp` workspace member.
- Error Handling Strategy:
  - Normalize connector failures into the existing retryable/non-retryable run model, persist invocation metadata before returning control to runtime, and gate low-trust external output before knowledge capture.
- Deferred Items:
  - Real MCP transport, credential handling, connector management UI, richer ToolSearch semantics, and non-`manual_event` trigger integration remain deferred.
- Non-goals Preserved:
  - No UI surfaces, no new trigger types, no approval-driven knowledge promotion, no vector retrieval changes, and no Org Graph work.

## Design Note

- Problem:
  - The tracked repository proves the governed runtime and real MCP transport baseline, but remote-hub still behaves like an unauthenticated thin shell and cannot distinguish remote auth/session failure from transport failure.
- Goal:
  - Add one minimum remote access boundary with persisted users, workspace membership, and JWT-backed sessions while preserving the existing runtime truth and shared client parity rules.
- Acceptance Criteria:
  - Remote access requires authenticated bearer tokens on protected routes.
  - Workspace-scoped routes enforce membership from JWT claims plus persisted membership checks.
  - Shared contracts expose auth-aware connection and session state to TypeScript consumers.
  - Desktop shared-client consumers can degrade to auth-required or token-expired state without new page families.
- Non-functional Constraints:
  - No second execution shell.
  - No app-local DTO truth.
  - No full tenant/RBAC rollout in this slice.
- MVP Boundary:
  - Bootstrap local user + password hash seed.
  - JWT access token with expiry.
  - Persisted session registry for revocation and logout.
  - Auth-aware hub connection status and error mapping.
- Layer Placement:
  - New access/auth persistence, hashing, JWT creation/validation, and membership lookup live in a dedicated Rust crate.
  - `apps/remote-hub` owns axum extraction, auth middleware, and HTTP mapping only.
  - `packages/hub-client` owns remote token injection and auth-failure normalization only.
  - `apps/desktop` owns auth-aware status display and read-only degradation only.
- Module Boundaries:
  - Runtime remains untouched as authority execution shell.
  - Auth crate owns user, membership, session, and token semantics.
  - Remote-hub app validates auth context before calling runtime methods.
- Inputs:
  - Login credentials.
  - Authenticated bearer token.
  - Existing workspace/project/run identifiers on remote routes.
- Outputs:
  - Login/session DTOs.
  - Auth-aware connection-state DTOs.
  - HTTP 401/403 failures with structured auth failure shape.
- Error Handling:
  - Missing/invalid bearer token maps to auth-required.
  - Expired token maps to token-expired.
  - Valid token without workspace membership maps to forbidden.
  - Transport disconnect remains separate from auth state.
- Tech Stack Decision:
  - Use `jsonwebtoken` for JWT signing/verification and a password-hash crate for local password verification.
- Test Strategy:
  - Red-first integration tests for login, protected-route auth failures, workspace mismatch, logout/revocation, and authenticated happy path.
  - TS tests for schema parsing and remote client auth failure normalization.
- ADR Needed:
  - Only if the new access/auth crate establishes durable placement rules not already covered by existing owner docs.

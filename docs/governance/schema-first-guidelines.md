# Schema First Guidelines

## 1. 目的

本文件定义 Octopus 的 Schema First 开发约束。  
目标是让跨语言共享契约先于实现落地，避免 Rust 与 TypeScript 各自演化、接口漂移、事件失配、状态机失真。

---

## 2. 核心原则

### 2.1 共享契约先于实现

凡是以下对象，默认先定义或更新 `schemas/`，再进入实现：

- Commands
- Queries
- Events
- 公共 DTO
- 核心枚举
- 状态机关键状态与转移
- 跨 Rust / TypeScript 共同依赖的对象

### 2.2 `schemas/` 是唯一事实源

`schemas/` 中定义的是跨语言共享契约的唯一事实源。  
`crates/` 与 `packages/` 只能消费或包裹这些契约，不得私自定义平行版本。

### 2.3 不把所有对象都塞进 schema

仅当对象具有以下特征时，才进入 `schemas/`：

- 跨语言共享
- 公共接口依赖
- 长期稳定
- 需要在多个模块中达成一致

纯粹的单语言内部实现对象，不应强行进入 `schemas/`。

---

## 3. 哪些对象必须 Schema First

以下对象默认必须先落 schema：

## 3.1 运行时核心对象

- Run
- Task
- Automation
- ApprovalRequest
- Artifact
- KnowledgeCandidate
- Shared Knowledge 入口对象

## 3.2 协议类对象

- Command
- Query
- Event
- Response DTO
- Error DTO（如对外公开）

## 3.3 状态机相关对象

- 状态枚举
- 状态迁移事件
- 恢复 / 重试 / 终止等关键动作语义

## 3.4 跨端共享对象

- Desktop 与 Remote Hub 共同依赖对象
- Rust 与 TypeScript 都要消费的结构
- 审批、预算、授权解释结果

---

## 4. 哪些对象不必进入 schema

以下对象通常不必进入 `schemas/`：

- 单语言内部 helper
- 前端临时页面状态
- 视图层纯展示对象
- Rust 内部仓储细节
- ORM 模型细节
- 缓存层内部结构
- 临时实验对象

如后续演进为跨语言共享对象，再迁移进入 `schemas/`。

---

## 5. Schema 设计要求

## 5.1 命名要求

- 命名必须稳定、可搜索、语义明确
- 避免使用模糊名如 `Data`、`Info`、`Payload` 作为总称
- 状态枚举命名应能表达生命周期语义
- Event 命名应使用过去式或明确动作语义

## 5.2 字段要求

- 字段必须有明确语义
- 必填与可选边界必须清晰
- 避免用万能 map 代替结构化字段
- 时间、ID、状态等关键字段必须规范命名
- 对不变量有帮助的字段应尽量明确表达，而不是依赖注释补救

## 5.3 版本与兼容性要求

每次 schema 变更必须回答：

- 是否向后兼容
- 是否影响已有消费者
- 是否需要迁移
- 是否需要并行支持旧版本一段时间

## 5.4 状态机要求

涉及状态机时，schema 不仅要定义状态，还要定义：

- 核心状态集合
- 允许的关键迁移
- 关键触发事件
- 非法状态组合的约束

---

## 6. Schema 变更流程

推荐流程：

1. 明确变更目标与影响范围
2. 在 `schemas/` 中新增或修改契约
3. 补充兼容性说明
4. 补充受影响消费者说明
5. 更新生成代码或消费层封装
6. 更新 Rust / TypeScript 实现
7. 补充契约测试或消费测试
8. 更新相关文档或 ADR

---

## 7. 何时必须补兼容性说明

满足以下任一情况时，必须明确写兼容性说明：

- 修改公共字段
- 修改字段语义
- 增删状态枚举
- 修改 event 含义
- 修改 command / query 形状
- 影响多个语言消费者
- 影响持久化或历史数据读取

---

## 8. 代码生成与手写代码边界

### 8.1 生成代码适合做什么

- 基础类型
- DTO
- 枚举
- 协议封装
- 基础序列化 / 反序列化绑定
- 基础 client / model scaffold

### 8.2 手写代码适合做什么

- 领域逻辑
- 校验逻辑
- 复杂不变量
- 编排逻辑
- 业务流程
- 错误处理
- 映射封装
- 兼容性适配

### 8.3 禁止事项

- 在生成代码中手写核心逻辑
- 让手写代码反向成为契约事实源
- 生成代码与手写代码彼此覆盖真相
- 不清楚边界就随意扩展生成模型

---

## 9. 测试要求

Schema 相关变更至少应考虑：

- 契约测试
- 序列化 / 反序列化测试
- 枚举兼容性测试
- 消费侧映射测试
- 关键状态迁移的契约测试

如果 schema 变更影响公共行为，则不得只测实现，不测契约。

---

## 10. 与 AI 开发的配合规则

AI 在做以下事情前必须先问自己：

- 这是共享对象还是内部对象
- 是否需要进入 `schemas/`
- 是否会被 Rust 与 TypeScript 同时消费
- 是否会影响公共接口或状态机
- 是否需要兼容性说明
- 是否需要 ADR

如答案不明确，不得直接绕过 schema 去实现。

---

## 11. 常见反模式

- 先写实现，最后再“补一个 schema”
- Rust 与 TypeScript 各写一套近似但不一致的对象
- 让 UI 展示模型偷偷变成公共协议真相
- 事件名、状态名、字段名在不同语言各叫各的
- 为图省事用 `map[string]any` 或 `Record<string, unknown>` 替代清晰结构
- 修改了 schema 却没有兼容性说明

---

## 12. 一句话总纲

在 Octopus 中，跨语言共享契约不是实现后的附属品，而是实现前必须先冻结的边界事实。

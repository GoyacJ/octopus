## Design Note

- Problem:
  - GA promises `MCP event` triggers, but the tracked repository only proves MCP-backed execution, not event ingress.
- Goal:
  - Add one governed MCP-event dispatch path that validates the source server and event selector before projecting into the existing automation delivery shell.
- Acceptance Criteria:
  - `mcp_event` trigger metadata records the selected MCP server and event match information.
  - Runtime rejects unknown servers and event mismatches explicitly.
  - Accepted MCP events create or reuse one deduped trigger delivery and automation run.
  - Low-trust artifact and knowledge-gate behavior remains unchanged.
- Non-functional Constraints:
  - No real provider transport or credential semantics.
  - No second interop execution shell.
- MVP Boundary:
  - Runtime ingress only against registered `McpServer` records.
- Layer Placement:
  - MCP-event metadata contracts stay in `schemas/runtime`.
  - Server validation and dispatch stay in `crates/runtime` with existing `interop-mcp` store access.
  - No app/package surface work.
- Module Boundaries:
  - `AutomationIntake` owns trigger persistence.
  - `Slice2Runtime` owns MCP-event dispatch validation and projection.
  - `RunOrchestrator` and the interop gateway remain responsible for downstream execution and knowledge gating.
- Inputs:
  - `trigger_id`
  - `server_id`
  - `event_name`
  - `dedupe_key`
  - opaque JSON payload
- Outputs:
  - A `TriggerDeliveryReport` or explicit runtime rejection.
- State Transitions:
  - `TriggerDelivery` state transitions remain unchanged.
- Error Handling:
  - Unknown server and event mismatch reject early.
  - Duplicate deliveries reuse the prior projection.
- Tech Stack Decision:
  - Reuse existing runtime + `interop-mcp` store APIs.
- Visual Framework Impact:
  - None.
- Human Approval Points:
  - None.
- Reused Components:
  - Shared trigger substrate, delivery projection, MCP-backed execution, and low-trust knowledge gating.
- New Abstractions:
  - Minimal MCP-event trigger matching helper.
- Trade-offs:
  - The slice proves governed ingress only and intentionally avoids pretending that real MCP transport is solved.
- Test Strategy:
  - Add failing runtime tests first for accepted, rejected, duplicate, and low-trust-gate-preserving flows.
- ADR Needed:
  - No.

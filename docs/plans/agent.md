# 数字员工/数字团队统一模型调整方案

## Summary

基于你的需求，目标模型定为：

- **工作区** 是“全量可见池”：展示 `工作区资产 + 内置模板`，代表当前工作区可用的全部数字员工/团队。
- **项目** 采用 **混合模型**：既允许有 **项目自有资产**（项目导入、项目创建、项目内复制产生），也允许在 **项目配置-智能体** 中选择 **工作区已有资产和内置模板**。
- **项目页展示生效集合**：项目数字员工/数字团队页展示 `项目自有 + 项目配置中已接入的工作区资产 + 项目配置中已接入的内置模板`，而不是展示全部候选。
- **会话选择器只基于项目已配置生效集合**，不再直接从工作区全量资产取数。
- **内置模板可直接分配到项目并在会话中使用**，但仍然 **不可编辑、不可删除**，只允许复制。
- **复制** 保留，且区分来源处理；**项目自有资产支持提升到工作区**。

当前设计里最大的冲突点是：  
项目资产现在同时有 `project links`、`project assignments`、`runtime enabled` 三套语义，需要收敛到一条主线，否则项目页、项目配置、会话选择器会继续不一致。

## Key Changes

### 1. 统一项目“生效资产集合”的定义

把项目中可见、可配置、可会话使用的资产统一到一个集合：

- `effectiveProjectAgents = projectOwnedAgents + assignedWorkspaceAgents + assignedBuiltinAgents`
- `effectiveProjectTeams = projectOwnedTeams + assignedWorkspaceTeams + assignedBuiltinTeams`

其中：

- `projectOwned*`：项目自有资产
- `assignedWorkspace*`：在“项目配置-智能体”里勾选的工作区资产
- `assignedBuiltin*`：在“项目配置-智能体”里勾选的内置模板

默认规则：

- **项目配置-智能体** 是工作区/内置资产接入项目的唯一入口
- 现有 `project links` 不再作为桌面端项目资产展示和会话选择的来源
- 若保留后端 `project link` 接口，视为兼容层；桌面端 UI 和 store 不再依赖它作为主模型

### 2. 页面职责重组

**工作区数字员工/团队页**

- 展示：`工作区自有 + 内置模板`
- 工作区自有：可编辑、可删除、可复制、可导入导出
- 内置模板：可查看、可复制；不可编辑、不可删除；默认可导出与否按实际需求控制，这次以“可作为项目正式资产来源”为前提，建议允许在项目导出时被物化导出

**项目配置-智能体**

- 候选列表展示：`工作区自有 + 内置模板`
- 保存的是“分配/接入关系”
- 项目自有资产不需要在这里“再接入”，默认属于项目生效集合
- 项目运行时 `enabledAgentIds / enabledTeamIds` 只在“生效集合”内做启用过滤

**项目数字员工/团队页**

- 展示“生效集合”，即：
  - 项目自有
  - 已接入的工作区资产
  - 已接入的内置模板
- 不展示未接入的工作区候选资产
- 来源对应行为：
  - 项目自有：可编辑、可删除、可复制、可导入导出、可提升到工作区
  - 工作区资产：只读；可“复制为项目资产”；不可在此直接删除源资产
  - 内置模板：只读；可“复制为项目资产”；不可编辑、不可删除

### 3. 导入导出与提升

继续使用现有 **bundle** 模型，不拆成 team 专属导入导出链路。

**工作区导入导出**

- 维持现有 bundle 机制
- 导入目标为 workspace scope
- 导出支持 agent/team 混选，team 自动带出成员 agent

**项目导入导出**

- 项目导入：导入后落为 **项目自有资产**
- 项目导出：导出 **项目生效集合**
  - 包含项目自有
  - 包含已接入的工作区资产
  - 包含已接入的内置模板
- 导出时需要把非项目自有资产物化进 bundle，而不是只导出引用关系

**提升到工作区**

- 仅对 **项目自有资产** 提供“提升到工作区”
- 语义上等同于“以 workspace scope 重新导入/复制一份”
- 提升成功后：
  - 工作区出现一份正式资产
  - 原项目资产保留，避免隐式迁移导致会话/引用断裂
- 是否自动替换项目引用为工作区版本：默认 **不自动替换**

### 4. 复制、编辑、删除权限矩阵

统一来源策略：

- **内置模板**
  - 可复制
  - 不可编辑
  - 不可删除

- **工作区资产**
  - 在工作区页：可编辑、可删除、可复制
  - 在项目页（作为接入资产出现）：只读，可复制为项目资产，不可在项目页删除源对象

- **项目自有资产**
  - 在项目页：可编辑、可删除、可复制、可提升到工作区

复制默认规则：

- 在工作区页复制：复制到工作区
- 在项目页复制：
  - 项目自有 -> 复制为新的项目资产
  - 工作区资产 -> 复制为项目资产
  - 内置模板 -> 复制为项目资产

### 5. 会话选择器收口

会话-智能体选择框改为只读“项目已配置且已启用”的资产集合：

- 数据源从当前 `workspaceAgents/workspaceTeams + enabled ids`
- 改为 `effectiveProjectAgents/effectiveProjectTeams + enabled ids`
- 因此会话中可选项自然覆盖：
  - 项目自有资产
  - 项目已接入的工作区资产
  - 项目已接入的内置模板

这一步是整个需求落地的关键，否则第 2 条和第 4 条仍然会脱节。

## Public Interfaces / Contract Changes

- 前端 store / selector 层新增统一的 `effectiveProjectAgents`、`effectiveProjectTeams` 派生集合，替代项目页和会话页对 `projectLinks` 的直接依赖
- 项目配置-智能体的候选源需扩展为 `workspace-owned + builtin-template`
- 项目导出接口的语义调整为“导出生效集合”，而不是仅导出项目自有或旧的 project-link 集合
- 需要显式支持“项目资产提升到工作区”的动作接口；优先复用现有 `copyToWorkspace` 能力，若现有接口只覆盖 builtin/特定来源，则补齐对项目自有资产的支持
- 旧 `project link` 接口保持兼容即可，但不再作为桌面端主流程的业务真相来源

## Test Plan

1. **工作区页**
- 展示工作区资产与内置模板
- 内置模板不可编辑、不可删除、可复制
- 工作区资产可编辑、可删除、可复制

2. **项目配置-智能体**
- 候选中同时出现工作区资产和内置模板
- 保存后项目生效集合正确更新
- 项目自有资产不需要重复勾选也能生效

3. **项目页**
- 仅展示生效集合，不展示未接入的工作区候选
- 项目自有、工作区接入、内置模板三类卡片动作正确区分
- 工作区接入/内置模板支持“复制为项目资产”

4. **导入导出**
- 工作区支持 agent/team bundle 导入导出
- 项目支持 agent/team bundle 导入导出
- 项目导出包含已接入工作区资产和已接入内置模板
- 导出 team 时自动包含成员 agent

5. **提升到工作区**
- 项目自有员工/团队可提升到工作区
- 提升后工作区出现新资产，项目原资产保留
- 提升后的资产可在其他项目配置中被选择

6. **会话选择器**
- 只出现“项目已配置且已启用”的员工/团队
- 已接入内置模板可以直接出现在选择器中
- 未接入或未启用的资产不会出现在选择器中

## Assumptions

- 采用你确认的 **混合模型**
- 项目页展示 **生效集合**
- 内置模板在项目配置中 **可直接分配并直接使用**
- 项目配置-智能体成为工作区/内置资产接入项目的唯一正式入口
- 项目 `project links` 视为旧模型兼容层，桌面端后续应逐步退场
- “提升到工作区”默认是 **复制提升**，不是迁移替换

## Delivery Note

- What Changed:
  - Added the Slice 2 task package and refined the shared contracts for `Task`, `Run`, `ApprovalRequest`, `CapabilityDescriptor`, `CapabilityBinding`, `CapabilityGrant`, `BudgetPolicy`, `InboxItem`, `Notification`, and `PolicyDecisionLog`.
  - Added the new Rust workspace member `crates/governance` for SQLite-backed capability, budget, and approval persistence/evaluation.
  - Extended `crates/observe-artifact` with inbox, notification, and policy-decision records plus persistence/query support.
  - Extended `crates/runtime` to evaluate governance before execution, persist pending approvals, resume approved runs, block rejected or denied runs, and expose query/report APIs for governance outputs.
  - Added the `0002_slice2_governance.sql` migration and new Slice 2 integration tests.
  - Updated repository entry docs so current tracked state reflects the new governed runtime slice.
- Why:
  - The GA blueprint requires Slice 2 to prove that Octopus execution is governed, not silent autonomy.
  - Keeping the slice local, Rust-only, and SQLite-backed preserves truthful scope while validating the formal approval lifecycle.
- User / System Impact:
  - Developers can now exercise a local governed execution loop with persistent approval, inbox, notification, and policy-decision records through Rust APIs and tests.
  - Existing Slice 1 runtime behavior remains available for low-risk allowed tasks and failure/retry/terminate paths.
- Risks:
  - The current governance model is intentionally narrow; later slices may still need to reshape capability and budget objects as Automation, MCP, or Shared Knowledge enter scope.
  - Notification behavior is persistence-only in this slice; there is still no delivery channel or app surface.
- Rollback Notes:
  - Roll back the Slice 2 migration, new `crates/governance` member, runtime/observation changes, and refined schemas together as one unit.
  - Partial rollback of schemas without the runtime and migration changes is not recommended.
- Follow-ups:
  - Start Slice 3 task packaging for Automation once the team is ready to extend non-manual entrypoints.
  - Decide whether the next transport-facing slice should expose the governed runtime through Tauri invoke, HTTP, or both.
  - Introduce PolicyDecisionLog or approval consumer generation only after the field set stabilizes across more slices.

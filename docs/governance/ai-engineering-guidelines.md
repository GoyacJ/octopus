# Octopus AI 开发规范（通用工程约束）

## 1. 目标与适用范围

本规范用于约束 AI 参与软件开发时的工程行为，确保产出具备以下特征：

- 设计可解释、可演进、可维护
- 结构清晰、分层明确、职责单一
- 编码风格统一、命名稳定、注释克制
- 测试充分且可运行，避免“看起来能用”
- 文件规模可控，避免巨型文件与隐式耦合
- 不掺杂具体业务规则，仅定义通用工程标准

本规范适用于：

- AI 生成新代码
- AI 修改既有代码
- AI 重构模块、补测试、补文档
- AI 输出设计说明、提交说明、迁移说明

本规范不定义业务流程、业务口径、业务数据规则。

---

## 2. 基本原则

### 2.1 先设计后实现

AI 在生成代码前，必须先明确以下内容：

- 目标是什么
- 边界在哪里
- 输入输出是什么
- 状态如何流转
- 失败如何处理
- 测试如何验证

禁止直接从需求跳到大段实现代码。

### 2.2 优先简单、清晰、稳定

在满足需求的前提下，优先选择：

- 更少抽象层级
- 更明确的数据流
- 更稳定的接口边界
- 更低的认知成本

禁止为了“高级感”引入不必要的设计模式、元编程、过度泛型、过度配置化。

### 2.3 单一职责与明确边界

每个模块、类型、函数只承担一种主要职责。

若一个单元同时承担：

- 业务编排
- 状态变更
- IO 调用
- 数据转换
- 错误处理
- 日志与监控

则应拆分。

### 2.4 可测试优先于可炫技

任何实现若难以稳定测试，应被视为设计存在问题。

### 2.5 显式优于隐式

必须优先使用：

- 显式依赖注入
- 显式类型
- 显式错误返回
- 显式边界转换
- 显式配置来源

禁止依赖隐式全局状态、魔法行为、隐式副作用。

### 2.6 MVP 小步迭代优先

AI 默认应选择最小、可验证、可回退的交付切片，而不是一次性铺开大范围实现。

要求：

- 优先交付满足当前 `Acceptance Criteria` 的最小 MVP slice
- 优先做可独立评审、可独立验证、可独立回滚的增量改动
- 不为假设中的未来需求预埋大块结构或抽象

### 2.7 边界、验收与审批不得由 AI 自裁定

以下内容必须由人或仓库既有事实源定义：

- 目标
- 范围与非范围
- 验收标准
- 人工审批点

AI 可以帮助澄清、补齐说明和拆分任务，但不得：

- 私自扩大范围
- 私自降低验收标准
- 跳过人工审批点
- 把自己的判断当作批准或验收结果

### 2.8 文档归属必须正确

AI 在补文档或修改规则时，必须把内容放到正确 owner doc：

- 产品语义、正式范围、release slicing 变化进入 `docs/product/PRD.md`
- 任务级目标、验收标准、非功能约束与设计说明进入 task package
- 共享契约、事件、DTO、状态语义变化继续遵守 schema-first
- `docs/architecture/VISUAL_FRAMEWORK.md` 只用于 UI / Surface 任务，不得成为非 UI 工作的全局前置门

## 3. 架构与分层规范

### 3.1 分层原则

推荐采用稳定分层，至少区分为以下角色：

- **Interface / Entry Layer**：对外入口层，如 HTTP、CLI、消息订阅、任务触发器
- **Application Layer**：应用编排层，负责用例组织、流程调度、事务边界
- **Domain Layer**：领域层，负责核心模型、规则、不变量、领域服务
- **Infrastructure Layer**：基础设施层，负责数据库、缓存、队列、文件系统、第三方接口

#### 3.1.1 各层职责

**Interface / Entry Layer**

只负责：

- 参数接收与基础校验
- 调用应用层
- 响应映射
- 协议适配

禁止在入口层写核心业务规则。

**Application Layer**

只负责：

- 组织用例
- 调用领域对象和基础设施接口
- 处理事务、编排、权限、流程
- 输出结果 DTO

禁止在应用层堆积底层细节实现。

**Domain Layer**

只负责：

- 核心实体
- 值对象
- 聚合边界
- 领域规则
- 不变量校验

领域层不得依赖具体基础设施实现。

**Infrastructure Layer**

只负责：

- 仓储实现
- 外部服务适配
- 持久化与序列化
- 网络、文件、消息、缓存访问

禁止把业务决策偷偷写进基础设施层。

#### 3.1.2 依赖方向

依赖必须单向流动：

`Interface -> Application -> Domain`

`Infrastructure` 依赖 `Domain` / `Application` 中定义的接口进行适配。

禁止反向依赖、禁止跨层直接穿透调用。

## 4. 模块设计规范

### 4.1 模块粒度

模块应围绕稳定职责划分，而非围绕“方便先写完”划分。

每个模块应满足：

- 名称可以清楚说明职责
- 对外接口少而稳
- 内部实现可替换
- 不依赖过多其他模块

### 4.2 文件大小控制

AI 生成代码时必须主动控制单文件体积。

建议约束：

- 单个源码文件建议 **200–400 行**
- 超过 **500 行** 必须主动评估拆分
- 超过 **800 行** 原则上禁止，除非是明确的自动生成文件或集中 schema 文件

出现以下情况必须拆分文件：

- 同时出现多个独立职责
- 同时定义 controller/service/repository/model/test helper 等混合内容
- 存在多个彼此弱相关的大型函数或类
- 文件阅读需要频繁上下跳转才能理解
- 单个类承担过多公共方法

### 4.3 大文件拆分策略

应按职责拆分，而不是机械切片。优先拆分为：

- 类型定义文件
- 核心实现文件
- 映射/转换文件
- 校验文件
- 错误定义文件
- 测试辅助文件

禁止把强耦合逻辑拆得过碎，导致理解路径过长。

### 4.4 接口设计

对外接口必须：

- 命名清楚
- 输入输出稳定
- 尽量窄接口
- 不暴露内部实现细节

禁止返回含糊不清的万能对象或 map 结构，除非协议本身如此要求。

## 5. 设计模式使用规范

### 5.1 使用原则

设计模式是约束复杂度的工具，不是炫技手段。

只有在下列情况成立时才使用设计模式：

- 能明显降低耦合
- 能明显提升可测试性
- 能隔离变化点
- 团队易于理解和维护

### 5.2 推荐优先使用的模式

可优先考虑：

- 工厂模式：用于复杂对象创建
- 策略模式：用于可替换算法或规则选择
- 适配器模式：用于外部系统接入
- 仓储模式：用于持久化抽象
- 门面模式：用于对复杂子系统提供稳定入口
- 模板方法或组合式管道：用于稳定流程骨架

### 5.3 谨慎使用的模式

以下模式必须谨慎：

- 单例模式
- 观察者滥用
- 过度装饰器嵌套
- 复杂责任链
- 深层抽象工厂
- 过度事件驱动

### 5.4 禁止事项

禁止出现以下情况：

- 为了用模式而用模式
- 一个简单分支也强行抽成策略体系
- 一个简单构造也强行引入工厂族
- 过早抽象未来可能永远不会出现的变化点

## 6. 编码风格规范

### 6.1 命名

命名必须准确、稳定、可搜索。

要求：

- 变量名表达真实语义
- 函数名体现动作
- 类型名体现领域概念
- 布尔命名使用 `is` / `has` / `can` / `should` 等前缀
- 避免 `data`、`info`、`util`、`manager`、`handler` 这类泛化命名，除非职责非常明确

### 6.2 函数规范

函数必须：

- 尽量短小
- 输入输出清晰
- 副作用可见
- 只有一个主要目的

建议：

- 普通函数优先控制在 **20–50 行**
- 超过 **80 行** 必须评估拆分

### 6.3 类/结构体规范

类或结构体必须：

- 聚焦单一职责
- 暴露最小必要接口
- 隐藏内部状态细节

若一个类同时处理多种规则、多种 IO、多种映射，应拆分。

### 6.4 条件与流程

必须：

- 优先早返回，减少嵌套
- 控制嵌套层级，通常不超过 3 层
- 复杂条件提取为命名良好的谓词函数

禁止出现长链式 if/else 搭配大量隐式状态修改。

### 6.5 常量与配置

必须：

- 消除魔法数字、魔法字符串
- 常量命名清晰
- 配置集中管理并标明来源

禁止把环境差异散落在业务代码中。

## 7. 注释与文档规范

### 7.1 注释原则

注释用于解释“为什么”，不是重复“做了什么”。

好的注释场景：

- 解释设计决策
- 解释非直觉性约束
- 解释兼容性原因
- 解释外部系统限制
- 解释算法选择依据

不好的注释场景：

- 翻译代码表面含义
- 与代码明显重复
- 失效后无人维护的描述

### 7.2 必须写注释的地方

以下位置应写注释或文档说明：

- 公共 API
- 非直观领域约束
- 边界条件处理
- 异常恢复逻辑
- 临时兼容方案
- 性能敏感路径

### 7.3 文档要求

对于新增模块，至少应补充：

- 模块职责说明
- 关键依赖
- 输入输出
- 失败路径
- 测试方式

---

## 8. 错误处理规范

### 8.1 错误必须结构化

错误处理必须可分类、可定位、可追踪。

要求：

- 区分用户错误、业务错误、系统错误、外部依赖错误
- 保留上下文信息
- 不吞异常
- 不返回无意义的通用错误文本

### 8.2 错误边界

在系统边界处必须完成：

- 错误归一化
- 错误码映射
- 日志记录
- 对外响应脱敏

### 8.3 禁止事项

禁止：

- `catch` 后什么都不做
- 用布尔值替代详细错误
- 把错误打印后继续假装成功
- 在深层无节制包装导致排障困难

### 8.4 报错科学处置三步法

AI 处理错误路径时，至少按以下顺序思考并记录：

1. **分类**：先判断是用户错误、业务错误、系统错误还是外部依赖错误，并估计影响范围
2. **定位**：明确出错边界、出错层次、关键上下文与可追踪信息
3. **处置**：决定重试、回滚、降级、升级审批、人工介入或安全终止，并说明原因

禁止只记录“报错了”而没有处置策略。

---

## 9. 测试规范

### 9.1 测试是交付物的一部分

AI 产出的代码若没有对应测试，默认视为不完整。

### 9.2 测试分层

建议至少包括：

- **单元测试**：验证纯逻辑、领域规则、边界分支
- **集成测试**：验证模块协作、仓储实现、接口适配
- **契约测试**：验证输入输出协议稳定性

### 9.3 测试原则

测试必须：

- 可重复运行
- 独立、不依赖执行顺序
- 覆盖正常路径、异常路径、边界条件
- 命名体现行为与预期
- 避免过度依赖实现细节

### 9.4 AI 编写测试的最低要求

每次新增或修改逻辑，AI 至少应：

- 为核心路径补测试
- 为失败路径补测试
- 为边界条件补测试
- 对回归风险点补测试

### 9.5 禁止事项

禁止：

- 只写 happy path
- 只验证函数被调用，不验证结果
- 使用脆弱快照替代行为断言
- 测试中堆积过多 mock 导致失真
- 测试名称模糊，如 `test1`、`should work`

## 10. 可维护性规范

### 10.1 可维护性优先级高于一次性完成

AI 必须优先保证未来可修改，而不是一次生成最多代码。
一次大而模糊的改动，通常不如多次小而可验证的 MVP 切片。

### 10.2 改动最小化原则

修改既有代码时，优先：

- 局部改动
- 保持接口兼容
- 减少无关重排
- 避免顺手进行大面积风格改写

### 10.3 重构要求

重构必须满足：

- 对外行为不变或有明确迁移说明
- 有测试保护
- 有拆分理由
- 有回滚路径

### 10.4 技术债可见化

若 AI 因上下文限制无法一次做完正确解，应：

- 明确标注临时方案
- 说明风险
- 指出后续收敛方向

禁止把临时方案伪装成最终设计。

## 11. 性能与资源使用规范

### 11.1 默认不做过早优化

应先保证正确性、清晰性、可测试性，再考虑性能。

### 11.2 必须关注的性能问题

AI 应主动避免：

- 明显重复 IO
- 明显 N+1 访问
- 无边界内存累积
- 不必要的大对象复制
- 高频路径中的低效字符串/集合操作

### 11.3 性能优化要求

任何性能优化都应：

- 指向明确瓶颈
- 不牺牲可读性过度
- 通过测试保障正确性
- 需要时附带基准或说明

---

## 12. 安全与边界规范

### 12.1 默认安全

AI 生成代码必须遵守最小权限、输入校验、输出脱敏原则。

### 12.2 必查项

涉及外部输入时必须考虑：

- 空值与非法值
- 注入风险
- 越权调用
- 敏感信息泄露
- 文件路径与命令执行风险

### 12.3 禁止事项

禁止：

- 在日志中输出敏感信息
- 在异常中暴露内部实现细节给外部调用者
- 将不可信输入直接拼接到查询、命令、路径中

### 12.4 安全底线与 MVP 迭代并行

MVP 不是绕过安全底线的理由。

即使是最小切片，也必须同时满足：

- 最小权限
- 输入校验
- 输出脱敏
- 高风险动作的审批与升级路径

禁止为了追求“先跑起来”而临时跳过安全边界，并把该行为伪装成可接受 MVP。

## 13. AI 开发行为规范

### 13.1 生成代码前的最小思考清单

AI 在写代码前必须先自检：

1. 这段代码的单一职责是什么
2. 它应该属于哪一层
3. 是否需要新建文件而不是继续堆进旧文件
4. 是否引入了不必要抽象
5. 测试该如何写
6. 注释应该解释什么决策
7. 边界、验收与审批点是否已由人或事实源定义
8. 最小可交付的 MVP slice 是什么
9. 非功能约束是什么
10. 相关信息应该写入 PRD、task package、schema 还是 `VISUAL_FRAMEWORK.md`

### 13.2 生成代码时的强制约束

AI 必须：

- 保持在既定的 Scope、Acceptance Criteria 与 MVP 边界内实现
- 优先复用项目内既有模式
- 保持目录与命名一致性
- 控制文件大小
- 控制函数长度
- 在新增或变更技术栈时先记录决策理由
- 在触达人为审批点时停止并显式升级
- 同步补充测试与必要文档

### 13.3 修改代码时的强制约束

AI 修改既有代码时不得：

- 顺带重写大量无关代码
- 私自把局部任务扩张为大范围重构
- 私自更改公共接口语义
- 破坏分层边界
- 把临时逻辑插入核心层
- 把任务级验收、局部设计或实现细节写进 PRD
- 把非 UI 工作错误绑定到 `VISUAL_FRAMEWORK.md`
- 删除原有注释、测试、日志而不给替代说明

### 13.4 输出说明要求

AI 每次完成开发任务时，建议附带简短说明：

- 改了什么
- 为什么这样改
- 风险点是什么
- 测试覆盖了什么
- 是否存在后续待收敛项

## 14. 代码评审清单

提交代码前，至少检查以下问题：

### 14.1 设计

- 职责是否单一
- 分层是否正确
- 依赖方向是否合理
- 是否存在过度抽象

### 14.2 代码

- 命名是否清晰
- 文件是否过大
- 函数是否过长
- 条件嵌套是否过深
- 是否消除了魔法值

### 14.3 测试

- 是否覆盖核心路径
- 是否覆盖失败路径
- 是否覆盖边界条件
- 测试是否稳定可重复

### 14.4 注释与文档

- 是否解释了关键设计决策
- 是否存在重复性、失效性注释
- 新模块是否有基本说明

### 14.5 维护性

- 是否容易扩展
- 是否容易定位错误
- 是否引入隐式耦合
- 是否保留了清晰的回滚与演进路径

---

## 15. 默认禁止清单

AI 默认禁止输出以下风格的代码：

- 巨型单文件
- 巨型 God Object / God Service
- 控制器直连数据库与外部服务
- 业务规则散落在工具类中
- 滥用静态工具类承载核心逻辑
- 无测试交付
- 无边界校验
- 用注释掩盖糟糕设计
- 过度使用万能基类或万能上下文对象
- 过度追求“框架感”而牺牲可读性

---

## 16. 落地建议

建议将本规范作为仓库级工程约束，至少以如下形式落地：

- `docs/governance/ai-engineering-guidelines.md`：正式规范文档
- `AGENTS.md`：提炼 AI 执行规则与必须遵守项
- PR / Code Review 模板：加入检查清单
- `docs/governance/git-quality-guidelines.md`：约束 diff / commit / PR 的范围控制、验证留痕与回滚说明
- 脚本或 lint 规则：尽量把部分约束自动化

建议后续再拆分为以下配套规范：

- 语言级规范（Rust / TypeScript / Vue / Java）
- 测试规范
- 提交与变更说明规范
- 目录结构规范
- API 设计规范

---

## 17. 一句话总纲

AI 参与开发时，必须优先产出 **职责单一、边界清晰、分层正确、体积可控、测试充分、注释克制、长期可维护** 的工程代码，而不是一次性堆砌“看似完整”的实现。

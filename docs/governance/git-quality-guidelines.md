# Git Quality Guidelines

## 1. 目的

本文件定义 Octopus 中关于 Git 变更卫生与质量管控的规则。  
目标是让一次 change set 可审查、可验证、可回滚，避免把多个问题捆绑进同一次提交、PR 或交付说明中。

---

## 2. 适用范围

本规范适用于：

- commit
- PR
- 单次任务对应的 diff
- 交付前的变更集整理

本规范约束的是 **变更质量**，不是具体分支模型。  
若仓库后续引入专门的分支策略，应在不冲突前提下与本规范配合使用。

---

## 3. 核心原则

### 3.1 单一主题与原子变更

一次 change set 应尽量只承载一个明确目标或一个明确 MVP slice。

要求：

- 一个 diff 应能对应回 Task Definition 中的单一目标
- 无关修复、无关重构、无关格式化应拆开处理
- 同一次交付中的多个步骤，必须能解释为什么必须一起提交

### 3.2 范围受控

变更范围必须与已冻结的 `Scope`、`Out of Scope`、`Acceptance Criteria` 和 `MVP Boundary` 一致。

禁止：

- 借实现机会顺手扩大范围
- 把局部任务做成隐含的大重构
- 在未更新边界文档前继续扩张 diff

### 3.3 无关改动禁止

以下内容如非本次目标，不应混入同一次 change set：

- 大面积格式化
- 无关重命名
- 无关目录搬移
- 无关依赖升级
- 无关锁文件变化
- 无关生成产物刷新

如确需一起提交，必须在交付说明中明确理由。

### 3.4 验证证据留痕

任何“已验证”“已通过”“可合并”的结论，都必须附带可追溯证据。

至少应记录：

- 运行了什么验证命令
- 覆盖了哪些核心路径、失败路径、边界条件
- 还有哪些缺口

禁止没有 fresh verification evidence 就宣称通过。

### 3.5 回滚与兼容性透明

非平凡变更必须说明回滚与兼容性影响。

至少应回答：

- 哪些部分可以独立回退
- 哪些部分需要连同 schema / consumer 一起回退
- 是否存在临时兼容层
- 是否存在不建议部分回滚的原因

### 3.6 文档与 diff 对齐

当变更影响事实源时，必须同步更新正确 owner doc：

- 产品语义变化更新 PRD
- 任务级验收与设计更新 task package
- 共享契约变化遵守 schema-first
- UI / Surface 的视觉冻结事项更新或引用 `VISUAL_FRAMEWORK.md`

禁止只改实现或只改 PR 文案，而不更新真正的事实源。

---

## 4. 合并前最小要求

在允许进入可评审、可合并状态前，至少确认：

- change set 对应单一任务目标或单一批准后的批次
- diff 没有混入解释不清的无关改动
- 验证证据已经记录
- 非平凡改动已有 rollback notes
- 相关 owner docs、task package、ADR 已按需更新

---

## 5. 常见不合格模式

- 把行为变更和大面积格式化混在同一次改动里
- 把多个无关问题打包成“顺手一起改”
- 没有 fresh verification evidence 就写“测试通过”
- 部分回滚或临时兼容逻辑没有说明
- 修改生成文件或锁文件却无法解释来源

---

## 6. 与其他 owner docs 的关系

- `ai-phase-gates.md` 定义什么时候允许进入下一阶段
- `ai-engineering-guidelines.md` 定义 AI 的实现与边界行为
- `code-review-checklist.md` 把本规范转化为评审项
- `pull_request_template.md` 承载变更说明与验证留痕
- `change-delivery-guidelines.md` 约束最终交付说明

---

## 7. 一句话总纲

Git 质量不是“有没有提交”，而是一次变更是否单一、可审查、可验证、可回滚。

# Change Delivery Guidelines

## 1. 目的

本文件定义一次开发任务完成后应如何进行交付说明。  
目标不是重复提交 diff，而是明确告诉后续评审者与维护者：

- 这次改了什么
- 为什么这样改
- 风险点是什么
- 测试覆盖了什么
- 还有哪些临时方案或待收敛项

本文件与 `ai-delivery-templates.md` 中的 `Delivery Note` 配合使用。

---

## 2. 适用范围

以下场景都应提供交付说明：

- 新功能
- 缺陷修复
- 重构
- 契约变更
- 架构边界调整
- 测试补充
- 文档补充但影响工程执行方式

---

## 3. 交付说明的基本原则

### 3.1 讲结果，也讲理由

交付说明不能只写“改了哪些文件”，还必须说明：

- 为什么要这样改
- 为什么不采用其他方式
- 这次改动影响什么，不影响什么

### 3.2 必须透明暴露风险

如果存在以下情况，必须明确写出：

- 临时方案
- 已知缺口
- 未覆盖测试
- 兼容性风险
- 后续收敛建议

### 3.3 与任务规模匹配

小任务可以简写，但不能缺关键结论。  
中大型任务必须使用结构化说明。

---

## 4. 交付说明最小要求

任何非纯格式化改动，至少应说明：

- What Changed
- Why
- Risks
- Tests Included
- Follow-ups

---

## 5. 推荐标准结构

建议使用如下结构：

```md
## Delivery Note
- What Changed:
- Why:
- User / System Impact:
- Risks:
- Rollback Notes:
- Follow-ups:
- Docs Updated:
- Tests Included:
- ADR Updated:
- Temporary Workarounds:
```

---

## 6. 各字段填写要求

### 6.1 What Changed

说明本次实际完成了什么，建议按结果而不是文件列表描述。

好例子：

- 为 Run 创建链路增加了最小状态迁移与审计落点
- 将共享 DTO 迁移到 `schemas/`
- 为审批拒绝路径补充了失败测试与恢复说明

不推荐：

- 改了 8 个文件
- 新增了一些方法
- 调整了部分逻辑

---

### 6.2 Why

说明为什么采用当前方案。

应回答：

- 为什么需要这次改动
- 为什么按这种结构组织
- 为什么没有采用更复杂或更简单的另一种方式

---

### 6.3 User / System Impact

说明这次改动对外部行为、系统行为或维护方式的影响。

例如：

- 新增一个最小 approval 路径
- 调整共享契约字段，影响前端消费层
- 仅内部重构，不改变外部行为

---

### 6.4 Risks

必须说明主要风险点。

常见风险包括：

- 兼容性风险
- 回归风险
- 状态机遗漏风险
- 临时方案延续风险
- 未覆盖边界条件的风险

---

### 6.5 Rollback Notes

说明如何回滚或回退风险。

即使无法精确给出操作步骤，也应说明：

- 哪类变更可独立回退
- 哪类变更需连同 schema / consumer 一起回退
- 哪些变更不建议部分回退

---

### 6.6 Follow-ups

说明后续建议继续做什么。

例如：

- 需要补充契约测试
- 需要新增 ADR
- 需要把 app 内逻辑提取到共享层
- 需要移除临时兼容逻辑

---

### 6.7 Docs Updated

说明更新了哪些文档，或明确未更新且原因是什么。

---

### 6.8 Tests Included

说明补充了哪些测试，以及覆盖什么路径。

至少说明：

- 核心路径
- 失败路径
- 边界条件
- 是否仍有未覆盖点

---

### 6.9 ADR Updated

如本次涉及架构边界决策，必须说明是否新增或更新 ADR。

---

### 6.10 Temporary Workarounds

若存在临时方案，必须显式写出：

- 哪部分是临时方案
- 为什么暂时这样做
- 后续如何收敛

禁止省略。

---

## 7. 按任务类型的附加要求

## 7.1 新功能

除标准结构外，还应说明：

- 新能力边界
- 不做什么
- 最小验收依据

## 7.2 缺陷修复

应补充：

- Symptom
- Root Cause
- Regression Risk
- Tests Added

## 7.3 重构

应补充：

- 当前问题是什么
- 保持了哪些行为不变
- 拆分后结构有什么收益
- 哪些点未来还需继续收敛

## 7.4 Schema / Public API 变更

应补充：

- Compatibility impact
- Affected consumers
- Migration notes

---

## 8. 常见反模式

以下写法不合格：

- “一些优化”
- “修复部分问题”
- “改了很多地方，不便一一列举”
- “应该没问题”
- “测试通过”
- “后面再看”

这些写法既不传达结果，也不传达风险。

---

## 9. 与 Phase Gate 的关系

交付说明是 Phase 5 的核心产物之一。  
如果没有清晰交付说明，则本次任务不应标记为完成。

特别是以下情况，必须阻止“完成”状态：

- 没有风险说明
- 有临时方案但未标注
- 代码、文档、测试不一致
- 明显影响公共行为但未说明

---

## 10. 一句话总纲

一次改动的完成，不以“代码写完”为标准，而以“结果、理由、风险、测试与后续收敛项都已被清楚说明”为标准。

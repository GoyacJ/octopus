## Design Note

- Problem:
  - The repository already proves a governed local runtime through Shared Knowledge capture, but connector-backed execution and environment-lease semantics remain absent. `RunOrchestrator` still calls the deterministic execution engine directly, `crates/interop-mcp` is only a placeholder, and external outputs have no provenance or trust gate before knowledge writeback.
- Goal:
  - Deliver the minimum Slice 5 runtime that centralizes connector-backed invocation through one MCP gateway boundary, persists environment leases and invocation metadata, and preserves the existing governed run shell for both manual tasks and `manual_event` automations.
- Acceptance Criteria:
  - Connector-backed execution works through the same Run shell as built-ins.
  - Approval, retry, artifact, audit, and trace semantics remain compatible with Slice 1 through Slice 4.
  - Environment-lease lifecycle and stale-expiry behavior are persisted and test-verified.
  - Low-trust external output stops before knowledge-candidate creation.
- Non-functional Constraints:
  - No real MCP network transport in this slice; use fake/test-double execution only.
  - No UI or app-surface work.
  - No schema drift outside `schemas/`.
  - Preserve forward-only SQLite migration compatibility.
- MVP Boundary:
  - Manual tasks and `manual_event` automations only.
  - One connector-backed execution action shape plus deterministic fake gateway behavior.
  - Registry/query APIs exist for tests and local composition, not end-user configuration.
- Layer Placement:
  - `crates/execution` continues to own the formal action enum.
  - `crates/governance` owns capability catalog metadata and visibility queries.
  - `crates/interop-mcp` owns MCP server registry, environment leases, invocation persistence, health snapshot fields, fake gateway behavior, and trust/output-gate evaluation.
  - `crates/observe-artifact` owns artifact persistence and observation records, including new provenance fields on the artifact object.
  - `crates/knowledge` owns candidate/asset persistence and enforces knowledge-candidate shape updates.
  - `crates/runtime` remains the orchestration shell and composes governance, interop, observation, and knowledge managers.
- Module Boundaries:
  - `RunOrchestrator` remains the only execution shell and chooses between built-in deterministic execution and connector-backed gateway execution.
  - `interop-mcp` does not decide grant, budget, or approval; it executes only after runtime/governance have already allowed the run to continue.
  - Knowledge capture still happens after artifact persistence, but now checks artifact trust/provenance first.
  - Capability visibility queries live at the governance/runtime boundary, not inside the MCP gateway.
- Inputs:
  - Existing `CreateTaskInput`, `CreateAutomationInput`, and `DispatchManualEventInput`.
  - New connector-backed `ExecutionAction` payload with `tool_name` and structured `arguments`.
  - Seeded capability descriptors and fake MCP server registrations.
- Outputs:
  - `EnvironmentLeaseRecord`, `McpServerRecord`, `McpInvocationRecord`, and connector-backed artifacts with provenance metadata.
  - Runtime APIs for capability visibility, MCP registry seeding/query, invocation query, and lease lifecycle/query.
- State Transitions:
  - `EnvironmentLease`: `requested -> granted -> active -> released | expired | revoked`.
  - `Run`: unchanged formal states; connector-backed failures still surface as `failed` or `completed` through the existing run state machine.
  - `KnowledgeCandidate`: creation now depends on artifact trust gate in addition to existing project-space checks.
- Error Handling:
  - Fake MCP failures normalize into retryable or non-retryable execution failures before they touch run state.
  - Lease acquisition or gateway lookup failures become run failures with trace/audit evidence, not silent skips.
  - Low-trust artifacts persist, but knowledge capture records a trace explaining why Shared Knowledge writeback was gated.
- Tech Stack Decision:
  - Keep Slice 5 in Rust + SQLite only. No new runtime services or external dependencies beyond the new workspace crate.
- Visual Framework Impact:
  - None. Slice 5 does not touch UI or surface code.
- Human Approval Points:
  - None.
- Reused Components:
  - Existing `ExecutionEngine` for built-in deterministic actions.
  - Existing governance decision path, approval resolution path, automation delivery path, artifact persistence path, and knowledge capture/promotion path.
- New Abstractions:
  - `octopus-interop-mcp` as a dedicated interop boundary crate.
  - MCP registry/invocation/lease records and fake gateway executor.
  - Artifact trust-gate metadata consumed by knowledge capture.
- Trade-offs:
  - Slice 5 intentionally proves the execution boundary with fake MCP behavior rather than real transport. This keeps the adapter path truthful without implying remote connectivity is already solved.
  - Visibility queries stay minimal and project-scoped for now instead of implementing the full future ToolSearch service.
- Test Strategy:
  - Update schema-contract examples first.
  - Add a dedicated Slice 5 integration test file covering the new connector-backed runtime path and lease behavior.
  - Re-run the full workspace test suite for regression coverage.
- ADR Needed:
  - Yes. Slice 5 introduces a durable crate boundary and a lasting rule that external connector output must be gated before entering Shared Knowledge.

## Design Note

- Problem:
  - The tracked tree already proves remote-hub auth routes, remote transport parity, the desktop local host, and the route-split workbench, but desktop bootstrap still assumes a local host only path and `ConnectionsView` cannot authenticate or own session state.
- Goal:
  - Add one first-class desktop remote connection and session surface so the same desktop shell can choose local or remote hub mode, authenticate through the existing remote-hub auth routes, and surface remote auth degradation without forking the workbench IA.
- Acceptance Criteria:
  - Desktop can switch between local and remote modes from `ConnectionsView`.
  - Remote login / session / logout reuse the existing schema-owned auth DTOs.
  - The shell defaults to `/connections` when a remote profile exists without a valid in-memory session.
  - All post-login data loading continues through the existing `HubClient` and `hub` store.
- Non-functional Constraints:
  - No new shared schemas in this slice.
  - No persisted remote secret material.
  - No project discovery sidecar invented in desktop just to satisfy a default route.
- MVP Boundary:
  - One persisted connection profile.
  - One in-memory access token plus session summary.
  - One remote auth client added beside the existing remote data client.
  - One authority view for connect / disconnect state.
- Layer Placement:
  - `packages/hub-client` owns remote auth request/response wiring and error normalization.
  - `apps/desktop/src` owns connection profile persistence, in-memory session state, bootstrap route selection, and connection/session UI.
  - `apps/remote-hub` remains the auth route owner and should change only if client tests prove a mismatch in expected semantics.
- Module Boundaries:
  - `HubClient` remains the only task/run/automation/knowledge/governance surface abstraction.
  - `RemoteHubAuthClient` is auth-only and must not absorb task/run/query methods.
  - Desktop profile types stay app-local and do not enter `schemas/`.
  - Desktop must not invent a fake remote project identifier; when no project is known, the workspace-scoped workbench routes remain the safe entry path.
- Inputs:
  - Existing `HubLoginCommand`, `HubLoginResponse`, `HubSession`, and `HubAuthError` schemas and parsers.
  - Existing `createRemoteHubClient` token-injection and auth-error normalization utilities.
  - Existing desktop route split, `hub` store, read-only gating, and `ConnectionsView`.
- Outputs:
  - One auth-only remote client factory.
  - One desktop connection/profile store.
  - One bootstrap route-selection path that distinguishes local mode, authenticated remote mode, and unauthenticated remote mode.
  - One upgraded `ConnectionsView` with local/remote switching, remote login/logout, session summary, and auth-state messaging.
- State Transitions:
  - `local -> remote` persists the remote profile, clears any in-memory remote token, swaps the configured `HubClient`, and returns the shell to `/connections`.
  - `remote unauthenticated -> remote authenticated` stores the access token in memory, stores the session summary in memory, swaps the configured `HubClient`, refreshes connection status, and enters the workspace-scoped workbench.
  - `remote authenticated -> remote logged_out` revokes the session through remote-hub when possible, clears the in-memory token/session, refreshes the connection state, and returns to `/connections`.
  - `remote authenticated -> token_expired` keeps the shell in read-only mode and surfaces recovery through `ConnectionsView` without silently pretending the connection is disconnected.
- Error Handling:
  - Missing or invalid credentials surface the existing `HubClientAuthError` information.
  - Expired tokens remain distinguishable from transport disconnects in both `hub.connectionStatus` and the connection surface messaging.
  - Editing remote profile inputs clears the in-memory token/session so stale authorization cannot be reused across workspace/base-URL changes.
- Tech Stack Decision:
  - No new stack. Reuse the existing Rust + schema-ts + hub-client + Vue/Pinia/Tauri stack.
- Visual Framework Impact:
  - Yes. Slice 16 changes the `Connections` surface from a passive status card into an interactive connection authority view, but it keeps the existing workbench IA intact.
- Human Approval Points:
  - None.
- Reused Components:
  - `createRemoteHubClient(...)`
  - Existing hub auth parsers and DTOs
  - Existing desktop `hub` store read-only behavior
  - Existing workbench routes and shell layout
- New Abstractions:
  - `RemoteHubAuthClient` for auth-only remote routes.
  - One desktop connection/profile store for mode, profile persistence, and in-memory session state.
- Trade-offs:
  - Keeping the token memory-only avoids secure-storage scope creep, but remote login does not survive application restarts.
  - Defaulting authenticated remote mode into a workspace-scoped workbench route preserves truthful routing without inventing a project selector, but project-scoped routes remain unavailable until the shell learns a current project from an existing route or run.
- Test Strategy:
  - Red-first TS contract tests for auth client happy paths, token injection, and auth-aware failure normalization.
  - Red-first desktop tests for remote unauthenticated bootstrap, login success, logout reset, token-expired read-only behavior, and local/remote switch regression.
  - Keep the existing full-gate verification and add a mocked desktop remote smoke path.
- ADR Needed:
  - No, unless implementation forces a durable new repository rule for desktop credential handling beyond this slice.

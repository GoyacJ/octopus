## Implementation Summary

- Goal:
  - Generalize the automation trigger substrate from `manual_event` only to the full GA trigger set while keeping the current one-automation-one-trigger boundary.
- Files Added:
  - `crates/runtime/migrations/0006_slice6_trigger_expansion.sql`
  - `crates/runtime/tests/trigger_foundation.rs`
- Files Changed:
  - `schemas/runtime/trigger.schema.json` to define the discriminated union for `manual_event`, `cron`, `webhook`, and `mcp_event`
  - `crates/runtime/src/models.rs` to add `TriggerSpec`, typed trigger configs, and trigger-aware create / dispatch inputs
  - `crates/runtime/src/lib.rs` to preserve `dispatch_manual_event`, add `create_automation_with_trigger`, and route all trigger kinds through one shared delivery projection helper
  - `crates/runtime/src/services.rs` and `crates/runtime/tests/schema_contracts.rs` to persist trigger metadata, migrate forward from earlier slices, and keep schema truth aligned with runtime behavior
  - `Cargo.toml` and `crates/runtime/Cargo.toml` to register the runtime dependencies used by the follow-on trigger slices
- Files Removed:
  - None.
- Structure Decision:
  - Keep `Automation.trigger_id` unchanged, widen `Trigger` into a typed union, and funnel every trigger kind into one internal `dispatch_trigger_delivery(...)` path.
- Why This Structure:
  - GA needs four trigger types, but it should still have one governed delivery state machine and one compatibility story for existing `manual_event` callers.
- Reused Patterns:
  - Existing `TriggerDelivery -> Task -> Run` projection, reopen / retry semantics, and delivery-to-run state synchronization.
- New Dependencies:
  - The overall trigger-expansion program added `cron`, `chrono-tz`, and `sha2` to `octopus-runtime`; this foundation layer provides the widened typed substrate those trigger-specific wrappers use.
- Error Handling Strategy:
  - Reject trigger-type mismatches early, keep manual-event compatibility explicit, and preserve the existing delivery / run reopen and retry semantics instead of inventing a parallel trigger pipeline.
- Deferred Items:
  - Concrete `cron`, `webhook`, and `mcp_event` ingress behavior is delivered in the sibling Slice 6 through Slice 8 task packages, not inside this substrate slice by itself.
- Non-goals Preserved:
  - No multi-trigger-per-automation model, no second delivery state machine, no automation surface work, and no real MCP transport work in this slice.

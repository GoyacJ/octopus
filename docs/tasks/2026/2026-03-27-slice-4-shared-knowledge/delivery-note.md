## Delivery Note

- What Changed:
  - Added the Slice 4 task package and ADR 0004 for the dedicated knowledge crate and the project-scoped Shared Knowledge closed loop.
  - Refined shared contracts for `KnowledgeSpace`, `KnowledgeCandidate`, `KnowledgeAsset`, and `KnowledgeLineageRecord`, and added dedicated candidate / asset status schemas.
  - Added the new Rust workspace member `crates/knowledge` plus SQLite migration `0004_slice4_shared_knowledge.sql` for knowledge spaces, candidates, assets, capture retries, and lineage records.
  - Extended `crates/observe-artifact` with knowledge-lineage persistence plus Slice 4 audit / trace constants.
  - Extended `crates/runtime` with `ensure_project_knowledge_space(...)`, `promote_knowledge_candidate(...)`, `retry_knowledge_capture(...)`, knowledge-aware `RunExecutionReport`, pre-run recall, post-run candidate capture, and query helpers.
  - Added integration coverage for candidate capture, promotion + recall, and non-blocking retry recovery, and updated repository entry docs to reflect the new tracked state.
- Why:
  - The GA blueprint requires Slice 4 to prove Shared Knowledge can participate in the governed runtime loop without bypassing Task / Run semantics or waiting for later UI and transport surfaces.
- User / System Impact:
  - Developers can now create a project knowledge space, capture candidate knowledge from successful runs, explicitly promote it into shared knowledge, and observe that later runs recall the promoted asset.
  - Existing manual-task, approval, retry, and `manual_event` automation behavior remains available and verified.
- Risks:
  - Retrieval is intentionally narrow and only matches exact `capability_id` inside one project knowledge space.
  - Promotion remains an explicit runtime API and does not yet route through approval or inbox flows.
- Rollback Notes:
  - Roll back the Slice 4 migration, runtime/schema changes, `crates/knowledge`, ADR 0004, and task-package docs together as one unit.
  - Partial rollback of contracts without the migration and runtime changes is not recommended.
- Follow-ups:
  - Add `knowledge_promotion` approval integration when governance/UI boundaries are ready.
  - Extend recall beyond exact capability matching once retrieval and ranking semantics are explicitly designed.
  - Revisit KnowledgeSpace views and Org Graph promotion in later knowledge slices.

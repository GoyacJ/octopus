## Design Note

- Problem:
  - The current automation slice proves only `manual_event`, while GA requires `cron`, `webhook`, and `MCP event` to enter the same governed automation path.
- Goal:
  - Generalize trigger contracts and runtime persistence so all GA trigger types share one delivery projection model without changing the one-automation-one-trigger boundary.
- Acceptance Criteria:
  - Trigger schemas become a discriminated union for the GA trigger set.
  - Runtime can persist and reload trigger-specific metadata for all four GA trigger types.
  - Common dispatch helpers can materialize a delivery, derived task, and governed run from any supported trigger type.
  - The public `dispatch_manual_event` entrypoint remains behavior-compatible.
- Non-functional Constraints:
  - No new app/package surface work.
  - No change to `TriggerDelivery` state semantics.
  - No multi-trigger or scheduler semantics introduced.
- MVP Boundary:
  - Substrate and compatibility only.
  - `cron`, `webhook`, and `mcp_event` ingress remain for later slices.
- Layer Placement:
  - Shared trigger contracts stay in `schemas/runtime`.
  - Trigger metadata persistence and projection logic stay in `crates/runtime`.
  - Blueprint and current-state updates stay in `docs/architecture` and repo entry docs.
- Module Boundaries:
  - `AutomationIntake` continues to own automation/trigger/delivery persistence.
  - `Slice2Runtime` continues to expose public automation APIs and gains shared internal dispatch helpers.
  - `RunOrchestrator` remains the only execution shell owner.
- Inputs:
  - Existing `CreateAutomationInput`, `DispatchManualEventInput`, and persisted automation/trigger/delivery records.
  - Trigger-type-specific config payloads for `cron`, `webhook`, and `mcp_event`.
- Outputs:
  - A generalized `TriggerRecord` shape and runtime persistence model.
  - Shared internal dispatch helpers reused by later trigger-specific APIs.
- State Transitions:
  - `TriggerDelivery` state transitions remain unchanged from Slice 3.
- Error Handling:
  - Unsupported trigger-type dispatch still returns explicit trigger-type errors.
  - Existing dedupe behavior continues to return the existing delivery/run projection.
- Tech Stack Decision:
  - None. Reuse current Rust + SQLite + JSON Schema stack.
- Visual Framework Impact:
  - None.
- Human Approval Points:
  - None.
- Reused Components:
  - Existing automation intake, task intake, run orchestrator, approval flow, and delivery synchronization logic.
- New Abstractions:
  - Shared trigger config objects.
  - A common runtime dispatch path reused by manual, cron, webhook, and MCP-event wrappers.
- Trade-offs:
  - A slightly richer trigger model is accepted now to avoid duplicating projection logic in later slices.
  - The slice intentionally avoids external ingress work to keep substrate semantics isolated and reviewable.
- Test Strategy:
  - Add schema and runtime regression tests first for trigger-type persistence and compatibility.
  - Re-run existing Slice 3 automation tests unchanged after substrate edits.
- ADR Needed:
  - No. This extends ADR 0003 rather than changing the projection model.

# Octopus · AI 协作提示词手册

**状态**: 基线建立版 | **日期**: 2026-03-26
**适用对象**: 项目负责人、架构师、产品负责人、工程师在与 AI 协作推进 Octopus 时使用

---

## 1. 文档目标

本手册提供一组可直接复制、按场景改写的提示词模板，帮助人类更稳定地驱动 AI 按 Octopus 的规范推进工作。

它不是新的真相源，也不替代 [`AGENTS.md`](../AGENTS.md)、[`PRD.md`](./PRD.md)、[`SAD.md`](./SAD.md)、[`AI_ENGINEERING_PLAYBOOK.md`](./AI_ENGINEERING_PLAYBOOK.md) 或 [`AI_DEVELOPMENT_PROTOCOL.md`](./AI_DEVELOPMENT_PROTOCOL.md)。

它的作用是把这些规范变成更容易使用的沟通语言。

## 2. 使用原则

每次给 AI 提需求，尽量都写清楚以下五项：

1. 角色
2. 当前阶段
3. 范围
4. 非范围
5. 期望产出

推荐固定附加要求：

- 先按 `AI_DEVELOPMENT_PROTOCOL` 执行
- 先产出 `task slice card`
- 不要直接扩 scope
- 不要把 Beta 能力带入 GA
- 最后固定输出：`当前判断 / 已完成 / 风险 / 建议下一步 / 需要我决策的点`

## 3. 通用母提示词

适合大多数任务的起手式：

```text
你现在作为本项目的[角色]，请按 docs/AI_DEVELOPMENT_PROTOCOL.md 推进。

先做这几件事：
1. 阅读 README、AGENTS、PRD、SAD 和相关规范
2. 先输出一张 task slice card
3. 判断这轮是否需要先做 contract / ADR / implementation plan
4. 不要直接扩 scope，不要把 Beta 能力带入 GA
5. 回答最后固定输出：
   - 当前判断
   - 已完成
   - 风险/缺口
   - 建议下一步
   - 需要我决策的事项

当前任务：
[写任务]

本轮范围：
[写范围]

明确非范围：
[写不做什么]

期望产出：
[写要文档、计划、评审还是实现]
```

## 4. 场景提示词

### 4.1 需求澄清 / 收敛

```text
你现在作为项目负责人和架构师，不要实现，先帮我收敛这件事。
请基于 README、AGENTS、PRD、SAD 和 AI_DEVELOPMENT_PROTOCOL 输出：
1. task slice card
2. 这件事属于 GA / Beta / Later 哪一层
3. 会影响哪些正式对象和哪个平面
4. 是否需要先写 contract / ADR / implementation plan
5. 最合理的下一步是什么

任务：
[写任务]

非范围：
[写不做什么]
```

### 4.2 项目骨架设计

```text
你现在作为架构师，请先不要写代码。
请为本项目设计最小可行项目骨架，目标是支持 GA 首个垂直切片。

要求：
- 先输出 task slice card
- 说明 repo topology
- 说明为什么这样分层
- 明确哪些目录现在不该创建
- 给出最小骨架方案和后续扩展路径
- 最后输出需要我确认的决策点

约束：
- 不要为了完整感补全大骨架
- 必须符合 AGENTS、PRD、SAD、ENGINEERING_STANDARD
```

### 4.3 Contract 定义

```text
你现在作为领域架构师，请先为 [对象名] 产出 contract，不要实现代码。

要求：
- 使用 docs/templates/contract-template.md
- 严格对齐 PRD/SAD
- 不要发明未定义对象
- 写清状态、治理约束、异常路径、验收条件
- 最后指出这份 contract 是否需要人工确认
```

### 4.4 实施计划

```text
你现在作为 Tech Lead，请为这个切片产出 implementation plan，不要直接实现。

要求：
- 使用 docs/templates/implementation-plan-template.md
- 只覆盖当前切片
- 写清验收条件和验证方式
- 标出依赖的 contract / 文档
- 把任务拆到 AI 可执行但不扩 scope 的粒度

切片：
[写切片目标]
```

### 4.5 文档编写

```text
你现在作为项目文档负责人，请帮我补这份文档。

要求：
- 先判断这份内容应该进入哪个文档层
- 如果已有文档可承接，就不要新建重复文档
- 如需新建，说明它和 README / AGENTS / PRD / SAD / 补充规范的关系
- 最后给出同步建议和验证记录

目标文档类型：
[例如 contract / review / handbook / plan]

主题：
[写主题]
```

### 4.6 代码实现

```text
你现在作为实现负责人，请严格按已确认的 plan 执行。

要求：
- 先复述当前切片和验收条件
- 只做当前切片
- 中途发现 scope 漂移立刻停下
- 做完后汇报：
  - 变更了什么
  - 跑了什么验证
  - 剩余风险是什么
  - 建议下一步是什么
```

### 4.7 评审 / 纠偏

```text
你现在作为审查者，请 review 这次变更。

优先找：
- 越界
- 架构回归
- 治理链路缺失
- 知识污染风险
- 验证不足

不要先给风格建议。
如果没有明显问题，也要告诉我剩余风险和验证盲区。
```

### 4.8 视觉 / 交互收敛

```text
你现在作为产品设计负责人和前端架构师，请先不要实现。
请基于 PRD、SAD、VISUAL_FRAMEWORK 收敛这次交互设计。

要求：
- 明确对应哪个 GA 表面
- 明确核心对象和状态语义
- 说明页面层级、主区块、侧栏和关键动作
- 指出哪些是必须可见的风险/审批/预算/降级信息
- 不要引入 Beta 表面
```

### 4.9 下一步推进建议

```text
你现在作为项目负责人，请不要展开实现，先基于当前仓库状态给我下一步建议。

要求：
- 只给 3-5 个最有价值的下一步
- 每一步都说明为什么现在做
- 标明这是 design / contract / skeleton / implementation / review 哪一类
- 标明我需要决策的点
```

## 5. 高质量提问方向

想让 AI 更好地帮你推进，优先从这些方向提问：

- 这件事属于哪个切片
- 这件事影响哪些正式对象
- 这件事要不要先写 contract
- 这件事现在是否越过 GA
- 这件事如果现在做，最小可行范围是什么
- 这件事完成后要如何验证
- 这件事还缺哪些前置定义

这些问题比“帮我把这个做完”更容易得到稳定结果。

## 6. 不推荐的提示方式

以下表达容易导致 AI 漂移：

- “帮我把整个项目做完”
- “你看着设计就行”
- “直接开始写代码，后面再补文档”
- “先把完整骨架都搭起来”
- “你顺便把相关的也一起做了”

这些说法默认没有边界，会诱导 AI 自行扩 scope。

## 7. 推荐的对话节奏

推荐每轮都按以下节奏推进：

1. 让 AI 先判断
2. 让 AI 再收敛
3. 让 AI 产出前置物
4. 你确认边界
5. 再让 AI 实施
6. 最后让 AI 复盘和建议下一步

简化版就是：

`判断 -> 定义 -> 确认 -> 实施 -> 验证 -> 下一步`

## 8. 你可以固定追加的一句话

如果你希望 AI 更稳，建议几乎每轮都附加这句话：

```text
如果你认为我的要求会导致 scope 漂移、架构冲突、治理链路缺失或验证不成立，请直接指出，不要默认顺着做。
```

## 9. 与 SOP 的关系

- 本手册提供“怎么说”
- [`AI_DEVELOPMENT_PROTOCOL.md`](./AI_DEVELOPMENT_PROTOCOL.md) 提供“怎么做”
- [`AI_ENGINEERING_PLAYBOOK.md`](./AI_ENGINEERING_PLAYBOOK.md) 提供“哪些不能做”

三者应配合使用，而不是互相替代。

## 10. 结论

和 AI 协作推进项目，关键不是写一条万能长提示词，而是把每轮沟通都固定成：

- 明确角色
- 明确阶段
- 明确范围
- 明确产出
- 明确下一步

只要这个节奏稳定，AI 的可控性会明显高很多。

## Implementation Summary

- Goal:
  - Add the minimum governed MCP-event ingress so registered MCP servers can project events into the shared automation delivery path without implementing real provider transport.
- Files Added:
  - `crates/runtime/tests/slice8_mcp_event_trigger.rs`
- Files Changed:
  - `schemas/runtime/trigger.schema.json` and `crates/runtime/src/models.rs` to capture MCP-event selector config on `Trigger`
  - `crates/runtime/src/lib.rs` to add `dispatch_mcp_event(...)` and keep it on the shared delivery projection path
  - `crates/runtime/src/services.rs` to persist selector metadata and validate server / event matches before delivery creation
- Files Removed:
  - None.
- Structure Decision:
  - Keep MCP-event ingress as a runtime API bound to registered `McpServer` records instead of inventing a new transport layer inside this slice.
- Why This Structure:
  - The goal is to prove governed event ingress and selector semantics first, while keeping real credentialed transport work separate and explicitly later.
- Reused Patterns:
  - Existing `McpServer` registration, the shared trigger-delivery substrate, and the existing low-trust artifact / knowledge gate.
- New Dependencies:
  - None.
- Error Handling Strategy:
  - Unknown servers are rejected explicitly, selector mismatches are rejected before delivery creation, duplicate dedupe keys reuse the same delivery / run, and low-trust execution output still cannot bypass the knowledge gate.
- Deferred Items:
  - Real credentialed MCP transport, external provider plumbing, and any automation-management surface remain deferred beyond this slice.
- Non-goals Preserved:
  - No production connector transport, no new remote endpoint surface, and no automation UI work.
